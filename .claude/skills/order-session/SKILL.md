# 발표 순서 조정 (단원 리더용)

너는 이번 챕터의 단원 리더를 돕는다. 참여자들이 `chapters/chNN/README.md`에 각자 고른 핵심 개념을 적어두었다. 이 글들을 읽고 발표 순서를 조정하는 것이 목표다.

## 입력
- 챕터 번호: `$ARGUMENTS` (비어 있으면 사용자에게 "몇 챕터인가요?"라고 물어본다)
- 챕터 번호는 2자리로 zero-pad 한다. 예: `3` → `ch03`

## 절차 — 순서대로 실행

### 1. 오늘 날짜 확인
`date +%Y-%m-%d`를 실행해 오늘 날짜를 얻는다. (추정 금지)

### 2. 챕터 파일 읽기
`chapters/ch<NN>/README.md`를 읽는다. 파일이 없으면 거기서 멈추고 "해당 챕터 폴더가 아직 없습니다. 먼저 `_TEMPLATE.md`를 복사해 만드세요."라고 알린다.

각 참여자 섹션(`### <이름> — <개념>`)에서 다음을 추출한다: **이름 / 고른 개념 / 내용 리마인드 / 의미**.

### 3. 아직 안 올린 사람 체크
플레이스홀더(`<이름>`, `<고른 개념>`)나 빈칸으로 남아 있는 참여자를 찾아 명단으로 알린다.
→ "아직 안 올린 사람: A, B" (있을 때만)

### 4. 중복·겹침 점검
규칙상 각자 **서로 다른 개념**을 골라야 한다. 두 명 이상이 같거나 강하게 겹치는 개념을 골랐으면 플래그하고, 누가 어떻게 차별화하면 좋을지 한 줄씩 제안한다. (실제로 글을 고치진 말고 제안만)

### 5. 발표 순서 제안
추출한 개념들을 발표하기 좋은 순서로 배열한다. 우선순위:
1. **의존관계** — 기초 개념을 먼저, 그 위에 쌓이는 심화 개념을 뒤로
2. **책 흐름** — 책 본문 등장 순서와 크게 어긋나지 않게
3. **주제 인접** — 비슷한 주제끼리 붙여서

각 슬롯에 "왜 이 자리인지" 한 줄 근거를 붙인다.

### 6. README에 기록
챕터 README의 **`**리드**:` 줄 바로 아래**에 다음 섹션을 만들거나, 이미 있으면 통째로 교체한다:

```markdown
## 🗂️ 발표 순서 (리더 조정, <오늘 날짜>)
1. **<이름>** — <개념> · <한 줄 근거>
2. **<이름>** — <개념> · <한 줄 근거>
...
```

> 참여자가 쓴 "내용 리마인드 / 의미" **본문은 절대 수정·삭제하지 않는다.** 발표 순서 섹션만 추가/갱신한다.

### 7. 요약 출력
사용자에게 다음을 한 화면으로 보여준다:
- 발표 순서 (번호 매긴 목록)
- 중복 플래그 / 아직 안 올린 사람 (있으면)
- 참여자에게 보낼 공지 초안 한 줄 (예: "이번 주 발표 순서: 1.OO 2.OO 3.OO — 일요일 기준 조정했습니다")

## 마무리
변경(README 발표 순서 섹션)은 커밋하지 말고 **diff만 보여준 뒤** 사용자에게 "이대로 커밋·push 할까요?"라고 물어본다. (리더가 최종 확인 후 진행)
