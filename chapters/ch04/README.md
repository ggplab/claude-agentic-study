### 문종운 — 서브는 격리된 워커, 메인은 오케스트레이터 (분배·통합)
- **내용 리마인드**: 메인 에이전트는 오케스트레이터로서 **작업 분배 + 결과 통합** 둘만 하고, 서브에이전트는 메인과 격리된 환경에서 **단일 책임**으로 도는 워커다. 단일 책임·분리 기준 3가지(복잡도·컨텍스트 소모량·전문성)·파이프라인(순차·병렬)은 모두 이 한 축에서 파생된다.
- **나에게 어떤 의미였나**: 내 `/team` 하네스 전체가 이 한 문장의 구현이다 — 메인은 코드를 직접 짜지 않고 분배·통합만, 워커(`product-owner`·`architect`·`developer`·`qa-manager`)는 격리돼 1책임씩 맡는다. 무엇을 떼어낼지는 분리 기준 3가지로 판단하고, 한 줄 질문·잡일엔 하네스를 끼우지 않는다.

### 남서아 — 서브에이전트 vs 에이전트 팀: 언제 무엇을 쓸 것인가
- **내용 리마인드**: 서브에이전트와 에이전트 팀은 둘 다 일을 여러 에이전트에 나누지만, 통신과 조율 책임이 다르다. 서브에이전트는 오케스트레이터(메인)가 모든 워커를 통제하고 결과만 받아 합친다("Subagents only report results back to the main agent and never talk to each other"). 반면 에이전트 팀(실험적, v2.1.32+)은 팀원끼리 직접 메시지를 보내고 공유 태스크를 파일 Lock으로 제어하며 자가 조율한다("teammates share a task list, claim work, and communicate directly"). 공식 문서의 선택 기준: 결과만 받으면 되면 서브에이전트, 서로 도전하고 합의해야 하면 팀. 순차 작업·의존성 많은 작업엔 서브에이전트가 더 효과적. 서브에이전트는 `.claude/agents/*.md`로 정의하고 `--agent` 플래그로 호출하면, 프론트매터의 `tools`·`model` 경계가 실제로 강제되어 지시 출처가 단일화된다.
- **나에게 어떤 의미였나**: 클로드 플러그인을 만들면서 스킬 구조와 에이전트 설계가 결과물 품질을 크게 좌우한다는 점을 체감했다. 특히 테스트를 통과시키는 과정에서 구조를 단기적으로 다듬는 데 집중하다 보니, 좋은 결과와 좋은 구조가 반드시 같지 않다는 걸 느꼈다. 당장 초록불을 켜는 것과, 컨텍스트·권한·실패 경계가 정합한 구조를 만드는 것은 다른 문제였다. 이 경험을 통해 서브에이전트, 컨텍스트 관리, 실패 경계에 대한 기준을 다시 복습해야겠다고 생각했다. 특히 "팀이냐 서브에이전트냐"는 멋이나 규모의 문제가 아니라 — 워커끼리 대화할 필요가 있는가, 결과만 받으면 되는가라는 단 하나의 질문으로 갈린다는 걸, 공식 문서를 다시 읽고서야 언어로 정리할 수 있었다.
