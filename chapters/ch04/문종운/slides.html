§1 서브에이전트란

핵심 가치는 컨텍스트 격리

격리 없이 직접

  • 곁가지 작업이 메인 대화에 쌓임
  • 대량 읽기 → 메인 컨텍스트 팽창·오염

서브에이전트로 격리

  • 격리 환경에서 돌아 충돌↓
  • 메인엔 정제된 결과만 반환
실측 — 탐색 에이전트 ux-reviewer·claude-doctor는 read-only(Read,Glob,Grep)로 정의. 대량 스캔을 맡겨도 덤프가 메인에 안 쌓이고 "발견 항목"만 돌아온다
§2 오케스트레이터-워커

메인의 일은 분배 + 통합

메인 = 오케스트레이터 (분배 · 통합)
▼ 분배 ▼
워커 A
워커 B
워커 C
▲ 정제된 결과 통합 ▲
실측 — /team 하네스가 이 패턴. Plan→Design→Implement→Review→Complete를 전담 에이전트에 분배하고, .team/*.md 산출물을 메인이 통합. 메인은 코드를 직접 짜지 않는다

§4 지침의 균형

방향은 주되 판단 여지는 남긴다

  • 방향을 줄 만큼 구체적이되, 판단력을 발휘할 여지를 남기는 지침
  • 서브에 주는 초기 컨텍스트의 양, 서브가 수집한 컨텍스트의 관리가 결과를 좌우
실측 — developer.md: "설계 가정과 현실이 다르면 명시적으로 보고(임의 변경 금지)". 방향=설계서(위), 판단=불일치 보고(에이전트). 판단 비중 큰 일은 opus(architect·zero-trust-advisor), 절차 실행은 sonnet(developer)

실측 — chapter-deck이 2개 서브에이전트를 병렬 디스패치(메인 맥락 미상속)해 슬라이드를 만들었다. 독립적인 일은 한 메시지에서 동시에 띄우는 게 원칙

§7·§8 호출 · 재개

자연어로 부르고, agentId로 잇는다

호출

  • 자연어로 부르면 description 보고 자동 위임
  • 이름을 명시하면 더 정확히 지목

재개 (resume)

  • 서브는 새 인스턴스 → 이전 결과 모름
  • agentId 회수 → resume으로 맥락 유지
정정 — 메모의 @<이름>(agent)에서 (agent)는 실제 문법 아님. 이 하네스에선 Agent로 새로 띄우고, SendMessage에 id/이름을 줘야 맥락 유지 재개
§9 프론트매터

tools·skills로 능력 정의 — 단, 토큰

필드역할 / 주의
tools그 에이전트가 쓸 도구 허용 목록 (능력의 명시적 제약)
skills스킬 프리로드 → 절차 주입. 단 컨텍스트 급증 → 토큰 효율 따질 것
model티어 라우팅 (판단=opus / 실행=sonnet / 탐색=haiku)
실측 — developer.md: model: sonnet + tools: Read,Write,Edit,Glob,Grep,Bash(9개 중 유일한 쓰기 권한). 일관된 방법론은 CLAUDE.md 티어 규칙·품질 게이트로 통일 → 어느 에이전트든 같은 기준

§12 에이전트 팀 (실험적)

팀원끼리 통신, lock으로 race 차단

통신

  • config.json에 agent-id 배열
  • 이 파일을 보고 브로드캐스트 / 단일 메시지

공유 태스크

  • tasks 디렉토리의 작업을 수행
  • 파일 lock 선점 → 한 에이전트만 → race condition 차단
대비 — 서브에이전트: 메인이 결과 받아 오케스트레이션 / 에이전트 팀: 팀원이 직접 소통. 실측 analog는 /team sprint·work #N·integrate(멀티세션 병렬). 무거우니 기본은 서브에이전트
