## 1. 서브에이전트란 — 컨텍스트 격리

서브에이전트는 특정 작업에 특화된 에이전트 도구다. 메인 컨텍스트에 대한 맥락을 들고 작업할 수도 있으나, **메인과 격리된 환경에서 도는 것**이 핵심이다. 격리는 두 가지를 동시에 준다: ① 작업 충돌을 줄이고 ② 메인 컨텍스트에는 정제된 결과만 남긴다.

> **내 시스템 적용:** 탐색 전용 에이전트 `ux-reviewer`·`claude-doctor`는 `~/.claude/agents/`에 read-only(`Read, Glob, Grep`)로 정의돼 있다. UI 패턴 스캔이나 파이프라인 자가진단처럼 **대량 읽기**가 필요한 일을 이들에게 맡기면, 스캔 덤프가 메인 대화에 쌓이지 않고 "발견된 항목"만 압축돼 돌아온다. 이게 격리의 실효다.

## 2. 오케스트레이터-워커 패턴 — 분배와 통합

서브에이전트는 orchestrator–workers 패턴을 따른다. **메인 에이전트가 오케스트레이터**, 여러 서브가 워커다. 오케스트레이터의 핵심 작업은 단 둘 — **작업 분배**와 **결과 통합**이다. 워커들의 답을 받아 정제·처리하는 것이 메인의 몫이다.

> **내 시스템 적용:** `/team` 하네스가 이 패턴 그 자체다. Plan→Design→Implement→Review→Complete로 단계를 나눠 각 단계를 전담 에이전트(product-owner·architect·developer·qa-manager)에 **분배**하고, 각자 산출물(`.team/prd.md`·`.team/design.md`·`.team/code-map.md`)을 남기면 메인이 그걸 **통합**해 다음 단계로 넘긴다. 메인은 코드를 직접 짜지 않고 분배·통합만 한다.

## 3. 단일 책임 원칙

서브에이전트는 스킬과 똑같이 단일 책임 원칙을 지켜야 한다. 하나의 에이전트가 개발 계획도 세우고, 구현도 하고, 테스트까지 보완하면 **프롬프트가 복잡해지고** 결과 품질이 떨어진다.

> **내 시스템 적용:** `developer.md`는 책임을 "구현"으로만 못박는다 — 본문에 *"설계서에 없는 기능 추가, 최적화, 리팩토링 금지"*. 계획은 `product-owner`(PRD)·`architect`(설계), 검증은 `qa-manager`가 따로 맡는다. 한 명이 전 단계를 쥐지 않으니 각 시스템 프롬프트가 짧고 명확하게 유지된다.

## 4. 지침의 균형 — 방향 vs 판단, 그리고 컨텍스트 관리

좋은 서브에이전트 지침은 **방향을 줄 만큼 구체적이되, 판단력을 발휘할 여지는 남긴다.** 또한 서브에 전달하는 *초기 컨텍스트*를 얼마나 줄지, 서브가 *수집한 컨텍스트*를 어떻게 관리할지가 결과를 좌우한다.

> **내 시스템 적용:** `developer.md`의 *"설계 가정과 현실이 다르면 명시적으로 보고 (임의 변경 금지)"* 가 정확히 이 균형이다. 방향(설계서)은 위에서 주고, 불일치 상황의 **판단(보고 후 대기)** 은 에이전트에게 맡긴다. 판단 비중이 큰 일은 opus 티어(`architect`·`zero-trust-advisor`)로, 정해진 절차 실행은 sonnet 티어(`developer`)로 라우팅한다.

## 5. 파이프라인 — 순차·병렬 실행

서브에이전트는 파이프라인으로 엮어 **순차(체이닝)** 실행할 수도, **병렬** 실행할 수도 있다. 특히 프로젝트 컨텍스트를 조사·정제해 가져오는 일을 병렬화하면 **전체 작업 시간이 단축**된다.

> **내 시스템 적용:** `/team`은 단계 의존성이 있어 순차(plan→design→build→verify)로 흐르고, 사전조사처럼 의존성이 없는 일은 여러 탐색 에이전트를 **병렬** 디스패치한다. ch03 슬라이드를 만든 `chapter-deck`도 2개 서브에이전트를 병렬로 띄워 메인 맥락을 물려주지 않고 동시에 굴렸다. 독립적인 일은 한 메시지에서 동시에 띄우는 게 원칙이다.

## 6. 서브에이전트로 분리하는 기준 3가지

1. **작업 복잡도** — 멀티스텝으로 복잡하면 위임한다.
2. **컨텍스트 소모량** — 대규모 파일 읽기가 필요하면 위임한다.
3. **전문성 요구 수준** — 특정 도메인 전문성이 필요하면 위임한다.

세 기준의 공통 목적은 결국 **메인 에이전트의 컨텍스트를 오염시키지 않는 것**이다.

> **내 시스템 적용:** README 수십 개를 훑는 조사는 ②(컨텍스트 소모) → `Explore`/`general-purpose`에 위임. 보안 정합성 검토는 ③(전문성) → `zero-trust-advisor`. 다단계 기능 구현은 ①(복잡도) → `/team` 파이프라인. 반대로 한 줄 질문·잡일에는 하네스를 끼우지 않는다(분리 비용 > 이득).

## 7. 호출 방법

자연어 명령으로 호출된다. 예컨대 git 서브에이전트가 있다면 *"xxx 스테이징에 추가하고 커밋해줘"* 라고만 해도 클로드가 `description`을 보고 자동 위임한다. 더 정확히 지목하려면 에이전트 이름을 명시한다.

> **정정/보완:** 메모의 `@<에이전트이름>(agent)` 표기에서 `(agent)`는 실제 문법이 아니다. 기본은 **`description` 기반 자동 위임**이고, 명시 호출은 *에이전트 이름을 직접 지정*하는 형태다(이 하네스에선 Agent 도구의 `subagent_type`, 대화에선 이름 멘션). 핵심은 "이름을 못박을수록 의도대로 위임된다"는 점이다.

## 8. 결과 반환과 재개(resume)

서브의 작업 결과는 **메인 대화로 반환**되어, 메인이 활용하거나 후속 작업에 쓴다. 중요한 함정: 서브는 호출될 때마다 **새 인스턴스로 시작**하므로 이전 결과를 알지 못한다. 따라서 결과와 함께 **`agentId`를 받아 resume**으로 기존 에이전트가 맥락을 이어가게 해야 한다.

> **내 시스템 적용:** 이 하네스에서 `Agent`로 새로 띄우면 백지에서 시작하지만, `SendMessage`에 그 에이전트의 id/이름을 주면 **맥락을 유지한 채** 이어서 일을 시킬 수 있다. "새 호출 = 기억 없음, resume = 기억 유지" 라는 구분이 멀티턴 위임의 핵심이다.

## 9. 프론트매터 — tools·skills와 토큰 효율

프론트매터로 그 에이전트가 쓸 수 있는 `tools`와 `skills` 목록을 적절히 넣어 능력을 정의한다. 예를 들어 `git-master`에 내부적으로 `safe-rebase` 같은 스킬을 넣어두면 그 모델이 스킬을 적절히 활용한다. **단, `skills`를 프리로드하면 컨텍스트가 급격히 증가**하므로 토큰 효율을 따져야 한다. 또 서브가 **일관된 방법론**을 읽도록 해줘야 — 일관되지 않은 방법론을 주면 결과도 들쭉날쭉해진다.

> **내 시스템 적용:** `developer.md` 프론트매터는 `model: sonnet` + `tools: Read, Write, Edit, Glob, Grep, Bash`로, 9개 에이전트 중 유일하게 쓰기 권한을 부여한다(능력의 명시적 제약). 일관된 방법론은 `~/.claude/CLAUDE.md`의 티어 규칙(판단=opus / 실행=sonnet / 탐색=haiku)과 품질 게이트로 통일돼, 어느 에이전트를 띄워도 같은 기준으로 움직인다.

## 10. 훅 — PreToolUse · PostToolUse · Stop

서브에이전트 실행에는 세 시점의 훅이 걸린다: ① 도구 실행 **전**(`PreToolUse`) ② 도구 실행 **후**(`PostToolUse`) ③ 실행 **종료**(`Stop`). 이를 적절히 처리·기록하는 게 중요하다. 실패하면 *왜* 실패했는지 로그를 남기고 모니터링으로 관측해, 서브의 실패가 반복되지 않게 관리한다.

> **실측(이번 세션 산 증거):** 이 정리본을 쓰려고 `_TEMPLATE.md`를 Read 하자 `PreToolUse` 훅 `cbm-code-discovery-gate`가 실제로 **호출을 차단**했다(코드 탐색은 graph 도구 먼저 쓰라고). 확률적 지침이 아니라 코드가 결정론적으로 막는 게 훅이다.
>
> **보완:** 서브에이전트 맥락에선 **부모 세션** 쪽에 `SubagentStop` 훅도 있다 — 서브가 끝난 순간을 부모가 관측·기록하는 자리다. 메모의 "Stop"은 서브 자신의 종료, `SubagentStop`은 부모가 보는 완료로 구분하면 정확하다.

## 11. 에이전트 메모리

에이전트의 작업을 기록하고 재사용하고 싶다면 agent-memory 디렉토리를 활용한다.

> **내 시스템 적용:** `/team`은 단계별 산출물을 `.team/`(prd·design·code-map)에 파일로 남겨, 다음 단계 에이전트가 읽어 이어받는 **공유 메모리**로 쓴다. 새 인스턴스라도 이 파일들을 읽으면 직전 맥락을 복원한다(8절의 resume를 파일로 보완하는 방식).

## 12. 에이전트 팀 — 통신·공유 태스크·lock (실험적)

에이전트 팀은 실험적 도구다. 기존 서브에이전트가 *메인이 결과를 받아 오케스트레이션*하는 구조라면, 에이전트 팀은 **팀원끼리 직접 소통**하는 구조다. 가능한 이유:

- `teams/config.json`에 각 에이전트의 agent-id를 **배열**로 지정 → 팀원들이 이 파일을 보고 서로에게 **브로드캐스트**하거나 단일 메시지를 보낸다.
- task 처리도 `tasks` 디렉토리에 명시된 작업을 보고 수행하되, **race condition**을 막으려고 각 파일 접근에 **lock을 선점** → 한 에이전트만 그 task를 가져간다.

> **내 시스템 적용:** `/team sprint | work #N | integrate`가 멀티세션 병렬 개발 모드다. 여러 세션이 동시에 다른 이슈를 잡고(`work #N`), 마지막에 `integrate`로 합친다. 같은 이슈를 두 세션이 동시에 건드리지 않게 하는 게 메모의 **파일 lock = race condition 방지**와 같은 발상이다. 일상 작업엔 안 켜고, 병렬이 정말 필요할 때만 호출한다.

## 읽으며 생긴 질문과 내 결론

1. **"서브에이전트 vs 에이전트 팀, 언제 무엇을?"** → 결과만 정제해 받으면 충분하면 서브에이전트(오케스트레이터-워커). 팀원들이 *서로 상태를 주고받아야* 진척되는 일이면 에이전트 팀. 후자는 실험적이고 무거우니 기본은 서브에이전트다.
2. **"훅은 3개가 끝인가?"** → 서브 자신 기준 `PreToolUse`/`PostToolUse`/`Stop`이 맞지만, 부모가 서브 완료를 관측하는 `SubagentStop`까지 보면 그림이 완성된다(10절 보완).
3. **"resume이 없으면?"** → 매 호출이 백지 시작이라 직전 결과를 잃는다. 그래서 `agentId` 회수 또는 `.team/` 같은 파일 메모리가 사실상 필수다(8·11절).

## 이번 주 발표용 핵심 개념 1개

**"서브에이전트는 격리된 워커, 메인은 오케스트레이터 — 오케스트레이터의 일은 작업 분배와 결과 통합 둘뿐이다."**

- **고른 이유:** 임정은 4대 프리미티브를 "계획을 누가 쥐는가"로, 최지환은 "에이전트 정의 파일·프론트매터"로 잡았다. 겹치지 않게, 나는 **운영 관점의 오케스트레이션**(분배·통합과 그 파생물인 단일 책임·분리 기준 3가지·파이프라인)으로 4장을 꿴다.
- **내 작업에 닿는 지점:** `/team` 하네스 전체가 이 한 문장의 구현이다. 메인이 코드를 짜는 게 아니라 분배·통합만 하고, 워커는 단일 책임으로 격리돼 돈다 — 이 분업이 무너지면 프롬프트가 비대해지고 컨텍스트가 오염된다.
