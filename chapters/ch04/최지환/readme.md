# 최지환 · 4장 정리: 서브에이전트 / 에이전트 정의

> **메인 포인트**: 에이전트를 `.claude/agents/{name}.md` 파일로 *정의*하는 법 \
> — 프론트매터(필드)로 권한·호출 기준을, 본문(시스템 프롬프트)으로 역할·판단·소통을 적는다.
> 📎 대표개념 심화: [프론트매터 정리 (서브에이전트↔스킬)](frontmatter.md)

## 핵심 개념 요약

**서브에이전트 = Agent 도구로 호출되는 특화 AI 어시스턴트.** \
자체 컨텍스트 윈도우(독립 20만 토큰)·커스텀 시스템 프롬프트·할당된 도구를 가짐. \
핵심 가치는 **컨텍스트 격리** — 곁가지 작업이 메인 대화를 오염시키지 않고 결과만 반환.

$ 참조 : \
같은 주제를 **"정의 파일 = 설정 파일이 아니라 역할 계약서"** 관점으로 더 깊게 판다. \
agentcoding이 *무슨 필드가 있나*면, hanes는 *그 본문을 어떻게 잘 쓰나*.

* **정의 = 파일**:  YAML 프론트매터(메타데이터) + 본문(= 시스템 프롬프트).

* **저장 범위·우선순위**: \
  프로젝트 `.claude/agents/` > 사용자 `~/.claude/agents/` > 내장. 

* **패턴**: 오케스트레이터(메인)-워커(서브) \
  — 메인은 작업 분배·결과 통합, 서브는 **단일 책임 1개**씩. 

* **프론트매터 필드** : 필수 `name`·`description` 2개 / 선택 `tools`·`model`·`skills`·`disallowedTools`·`permissionMode`·`hooks`·`memory`. 

  * `tools`(허용목록)와 `disallowedTools`(거부목록)는 **택일**.

  * `permissionMode`(default/acceptEdits/dontAsk/bypassPermissions/plan)는 \
    `tools`보다 **상위** — `plan`이면 Write/Edit가 있어도 수정 불가.

$ 시스템 프롬프트의 뼈대 :
① 핵심 역할(범위) ② 작업 원칙(판단 기준) ③ 입출력 프로토콜(에이전트 간 API 계약) ④ 팀 통신 프로토콜(누구에게·뭘·언제) ⑤ 에러 핸들링(종료 조건·최대 재시도) ⑥ 협업(상·하류 위치) ⑦ 품질 자체 검증(제출 전 체크리스트).

* **호출**: 자연어로 부르면 클로드가 `description`을 보고 **자동 위임**. \
  `use PROACTIVELY`/`MUST BE USED` 넣으면 더 적극적으로 위임. 

* **에이전트 팀**(v2.1.32+, 실험): \
  서브에이전트=요청·응답(일회성), 팀=영속 협업(팀원 간 직접 통신·공유 태스크·파일 소유권). \
  환경변수 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` + 사용자 승인 필요.

* **CLI 동적**: \
  `--agent`(기존 정의 직접 실행, 독립 최상위 세션) \
   `--agents`(JSON 인라인 정의, `skills` 불가).
