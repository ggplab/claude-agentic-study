# CHAPTER 03 — 클로드 스킬

> 반복 절차·팀 지식을 `SKILL.md` 한 장으로 파일화하는 법. 점진적 작동으로 컨텍스트를 아끼고(3-1·3-2), `context: fork`로 스킬을 격리 실행 에이전트로 격상한다(3-3).

**리드**: 이정연

> ※ 챕터 제목은 메모의 절 제목("클로드 스킬의 개념과 구조")에서 따온 작업용 제목입니다. 실물 책 목차로 확정되면 갱신합니다.

## 📖 리드 핵심 요약 (이정연)

### 1. 스킬이란 — 절차를 파일화한 공유 지식 (3-1)

- 스킬 = `<이름>/SKILL.md` 디렉터리. `SKILL.md`만 필수, 스크립트·템플릿·레퍼런스 등 지원 파일은 복잡도에 따라 선택.
- 구성: **YAML 프론트매터(메타) + 마크다운 본문(절차)**. `name`은 행위, `description`은 기능 + **언제 불러야 하는가(트리거)**.
- 유형 3종 = **위치가 곧 범위**: 개인(`~/.claude/skills/`, 모든 프로젝트) · 프로젝트(`<레포>/.claude/skills/`, 그 레포 한정) · 플러그인.
- 저장소를 git에 커밋하면 팀원이 pull 받는 즉시 그 지식에 접근 → **빠른 온보딩**. ("컨텍스트 윈도우는 공용자산"의 출발점.)

> ❗ **정정**: 책/메모의 "200줄 이내" → **공식 권장은 500줄 이하**. "예산 100토큰" 같은 수치는 **공식 문서에 없는 추정치**. `name`·`description`은 공식 기준 **둘 다 선택**(description만 권장)이다.

### 2. 점진적 작동(progressive disclosure) — 3단계 지연 로딩 (3-1)

- **L1 메타데이터**(`name`+`description`): 세션 시작 시 **모든 스킬** 사전 적재.
- **L2 SKILL.md 본문**: 요청이 description과 매칭돼 **호출될 때만** 로딩.
- **L3 지원 파일·코드**: 모델이 **필요할 때 Read로만**.
- 단계마다 비용이 다르고 체계적으로 공개 → 유한한 컨텍스트 예산을 아끼는 설계.

### 3. 사용과 개발 (3-2)

- **스킬 조합**: 복잡한 작업은 여러 스킬을 조합. 유사 스킬의 동시 활성화를 막으려면 **고유·구체적 트리거 용어**를 description에.
- **개발 3단계**: 목적 정의 → SKILL.md(입출력 예시 2개 이상) → 지원 파일.
- **환각 방지 3종**: 불확실성 표현 허용 · 지식 범위 제한 · 출처 명시.
- **스크립트 우선**: 결정론적·비-토큰 작업은 스크립트로. **오류는 클로드에 넘기지 말고 스크립트 안에서 직접 처리**(실행권한·종속성·경로).
- **검증**: description 구체성 → 디렉터리 경로 → YAML 유효성 → `claude --debug`.

### 4. 고급 — `context: fork`로 격리 실행 (3-3)

- 재사용 패턴(프로젝트 특화 / 도메인 특화 / **범용 유틸리티**), Version History로 버전 관리, 메타 문서화(목적·예시·조합 관계·제한).
- **`context: fork`**: 스킬을 **부모 대화와 분리된 서브에이전트 컨텍스트**에서 실행 → 본문이 서브에이전트 프롬프트가 되고, **결과만 부모로 반환**. `agent`(Explore/Plan/general-purpose)는 `context: fork`일 때만 동작. `allowed-tools`와 조합하면 **격리 + 보안**을 동시에.
- 의미: 스킬을 **지식 모듈 → 독립 실행 에이전트 단위로 격상**. (단, "본문에서 Task로 서브에이전트 띄우기" ≠ "`context: fork`로 스킬 전체 포크" — 별개 메커니즘.)

### 이정연 — 스킬을 직접 만들며 배운 점 (개념 + 실습)
- **내용 리마인드**: 3장의 스킬은 "절차를 파일화한 공유 지식"이고, 점진적 작동·스크립트 우선·`context: fork`로 컨텍스트를 아끼며 격상된다. 개념에서 그치지 않고 **내 로보틱스 프로젝트에 실제 스킬 하나를 만들어** 개념을 손으로 검증했다.
- **실습으로 배운 점**: 손 URDF를 MuJoCo로 변환·검증하는 프로젝트 스킬 `urdf-to-mjcf`를 만들었다(`isaacgym_allegro_hand`, 68개 중 36 변환). 가장 크게 체감한 3가지 —
  1. **스크립트 우선**: 변환·검사를 스크립트로 빼고 오류(메시 경로 strip, ASCII STL 등)를 스크립트가 직접 처리하니, 클로드는 결과만 받아 토큰이 확 줄었다.
  2. **점진적 작동**: 절차(SKILL.md)와 지원파일(reference/EXAMPLES/scripts)을 나누니 본문이 가벼워졌다.
  3. **프로젝트 스킬 = 공용자산**: git에 올리니 팀이 클론만으로 같은 스킬을 쓴다 — 3-1 비유의 실증.
  - 발표 핵심으로 고른 `context: fork`(격리 실행)는 이번엔 보류했지만, 무거운 감사 단계를 떼어낼 다음 후보로 남겼다. (정리본: [이정연/README.md](이정연/README.md) · 실습 상세: [이정연/skill_practice.md](이정연/skill_practice.md))
- **LinkedIn**: 초안 확정(스크립트 우선·점진적 공개·공용자산을 손 68개 변환 실험으로 정리). 발행 후 링크 기록 예정.
