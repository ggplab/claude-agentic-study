## 3-1. 스킬의 개념과 구조

### 개념정리

- **핵심 비유 — "컨텍스트 윈도우는 공용자산"**: 유용한 지식·패턴을 공유 가능한 형태로 캡슐화한 것이 스킬이다. 스킬 저장소를 git에 커밋하면 팀원이 pull 받는 즉시 그 지식에 접근 → 빠른 온보딩.
  - (이 비유 자체는 임정이 발표 핵심으로 가져갔다. 나는 여기서 출발해 "그 캡슐을 실행 단위로 격상하는" 3-3 쪽으로 무게를 옮긴다.)
- **스킬의 유형 3종**: 개인 스킬(`~/.claude/skills/`, 모든 프로젝트) · 프로젝트 스킬(`<레포>/.claude/skills/`, 그 레포 한정) · 플러그인 스킬. **위치가 곧 범위**다.
- **디렉터리 구조**: `SKILL.md`만 필수. 스크립트·템플릿·레퍼런스 등 지원 파일은 복잡도에 따라 선택적으로 포함.
- **SKILL.md 구성**: YAML 프론트매터(메타) + 마크다운 본문(절차·유의사항·예시).
  - `name`: 스킬이 하는 행위를 나타내는 이름.
  - `description`: 기능 + **언제 불러야 하는가(트리거 조건)**. 자동 호출 판단의 근거라서 사실상 가장 중요한 필드.

> ❗ **정정 1 — 분량**: 메모엔 "200줄 이내 유지"라고 적었는데, **공식 권장은 500줄 이하**다(ch01 README §6·임정 README §3 모두 500줄 기준). single responsibility를 지향한다는 취지는 맞지만 숫자는 500으로 바로잡는다.
>
> ❗ **정정 2 — 필수 필드**: 메모는 `name`을 동명사로, `description`을 사실상 필수처럼 적었다. **공식 필드표 기준 `name`·`description` 둘 다 선택**이고 `description`만 *권장*이다(생략 시 본문 첫 문단을 description으로 씀, `name` 생략 시 디렉터리명 사용). 자동 호출을 원하면 description은 "실질적 필수"로 쓰게 되는 것뿐.

### 3단계 점진적 작동(progressive disclosure)

| 레벨 | 무엇이 | 언제 로딩 |
|------|--------|-----------|
| L1 메타데이터 | `name`+`description` | 세션 시작 시 **모든 스킬** |
| L2 SKILL.md 본문 | 절차 전문 | 요청이 description과 매칭돼 **호출될 때만** |
| L3 지원 파일·코드 | 스크립트·레퍼런스 | 모델이 **필요할 때 Read로** |

> ❗ **정정 3 — 토큰 수치**: 메모의 "예산 100토큰"은 **책 추정치이며 공식 문서엔 토큰 숫자가 없다**(임정 README도 동일 확인). "L1은 가볍고 L3는 사실상 무제한, 단계마다 비용이 다르다"는 정성적 사실만 취하고, 숫자는 추정으로 표기한다.

### 내 시스템 실측

- **내 `~/.claude/skills/`는 비어 있다 — 개인 스킬 0개.** (임정 47개와 정반대 출발점.) 즉 내가 지금 쓰는 스킬은 전부 **이 레포의 프로젝트 스킬**이다: `study-chapter`·`order-session`·`chapter-deck` (`.claude/skills/`). "위치가 곧 범위"가 그대로 보인다 — 이 셋은 이 스터디 레포의 README 양식·발표순서 규칙에 묶여 다른 데선 의미가 없으니 개인이 아니라 프로젝트 스킬로 있는 게 맞다.
- **지금 이 정리본을 만드는 과정 자체가 점진적 작동의 실증**이다: `study-chapter`의 L1(설명)은 떠 있지만, 본문 L2는 내가 작업을 시작할 때서야 로딩됐고, `_TEMPLATE.md` 같은 L3는 실제로 참조할 때만 Read로 끌어왔다.

## 3-2. 스킬 사용과 개발

### 개념정리

- **스킬 조합**: 복잡한 작업은 여러 스킬을 조합해 수행한다. 단, 유사 기능 스킬이 동시에 활성화될 수 있어 **각 스킬은 고유하고 구체적인 트리거 용어를 description에 박아야** 충돌이 안 난다.
- **커스텀 스킬 개발 3단계**: ① 목적 정의 → ② SKILL.md 작성(**입출력 예시를 2개 이상 다양하게** 넣어 모델이 패턴을 정확히 인지하게) → ③ 지원 파일 추가.
- **환각 방지 지침 3종** (스킬 본문에 명시):
  1. **불확실성 표현 허용** — 모르면 모른다고 말하게.
  2. **지식 범위 제한** — 참조 가능한 정보 범위를 좁힘.
  3. **출처 명시** — 정보의 출처를 밝혀 신뢰성 확보.
- **코드 실행 통합(스크립트 우선)**: 결정론적이고 토큰을 쓰지 않아도 되는 작업은 스크립트로. 핵심은 **오류 상황을 클로드에게 넘기지 말고 스크립트 안에서 직접 처리**(실행권한·종속성 설치·경로 슬래시 등). SKILL.md엔 사용법만.
- **테스트·검증**: ① description 구체성 확인 → ② 스킬 디렉터리 경로 확인 → ③ YAML 유효성 검사 → ④ `claude --debug`로 로딩 문제 확인.

### 내 시스템 실측 — 프론트매터로 본 "트리거·안전·인자"

레포의 세 스킬 프론트매터를 직접 읽어 비교했다(추정 아님):

| 스킬 | 핵심 필드 | 무엇을 말하나 |
|------|-----------|---------------|
| `order-session` | name + description + `argument-hint` | 읽기·정리 위주라 **자동 호출 허용**(차단 필드 없음) |
| `study-chapter` | + `disable-model-invocation: true` | **커밋·푸시하는 사이드이펙트 스킬 → 자동 호출 차단**(슬래시로만) |
| `chapter-deck` | + `disable-model-invocation: true` | 동일 — 생성·커밋하므로 안전장치 |

- 관찰: 내가 쓰는 스킬에서 `disable-model-invocation: true`는 **사이드이펙트(커밋/생성)가 있는 스킬에만** 붙어 있다. 메모엔 안 적힌 필드인데, "환각 방지"가 *내용*의 안전장치라면 이건 *실행*의 안전장치다.
- description의 트리거 구체성: 세 스킬 모두 "단원 리더가…", "챕터를 읽은 뒤…", "챕터 정리 메모를 받아…"처럼 **호출 상황을 문장으로** 적어 서로 겹치지 않게 했다 — 3-2가 말한 "고유 트리거 용어"의 실물.

## 3-3. 고급 사용과 최적화 — 그리고 내 발표 핵심

### 개념정리

- **재사용 가능한 스킬 패턴**: 특정 프로젝트/스택에 종속되지 않고, description에 충분한 트리거를 담고, 명확한 체크리스트로 다른 스킬과 조합되기 쉬운 형태. 패턴 종류는 **프로젝트 특화 / 도메인 특화 / 범용 유틸리티**(코드리뷰·문서생성·테스트·리팩터링 — 재사용성 최상).
- **버전 관리**: SKILL.md 안에 **Version History 섹션**을 두어 변경 이력 문서화 권장.
- **메타 문서화**: 스킬 문서엔 ① 목적·적용 범위 ② 실제 사용 예시 ③ 다른 스킬과의 관계(조합 패턴) ④ 알려진 제한·주의를 담아라.
- **디버깅**: `claude --debug`로 로딩 과정 문제 확인.

### ⭐ 스킬의 격리 실행 — `context: fork` (발표 핵심)

- `context: fork`가 설정된 스킬은 **부모 대화로부터 완전히 분리된 서브에이전트 컨텍스트**에서 실행된다.
- **스킬 본문이 곧 서브에이전트를 구동하는 프롬프트**가 되고, 서브에이전트는 부모 대화 히스토리에 접근하지 못한다. 작업이 끝나면 **결과만 부모 대화로 반환**된다.
- `agent` 필드는 **`context: fork`일 때만** 동작하며 서브에이전트의 모드를 정한다: `Explore`(코드베이스 탐색) · `Plan`(컨텍스트 수집·계획 수립) · `general-purpose`(기본값, 제약 없음).
- **`context: fork` + `allowed-tools`** 를 조합하면 **격리(컨텍스트)와 보안(도구 권한)을 동시에** 얻는다.
- 의미: 스킬을 **단순 지식 모듈 → 독립 실행 가능한 에이전트 단위로 격상**시키는 메커니즘. 서브에이전트의 격리를 누리면서도 스킬 고유의 **자동 발견·동적 활성화 장점은 그대로** 유지한다.
- 적합한 경우: 대규모 코드베이스 분석 · 세션 격리가 필요한 워크플로 · 도구 전문화 기반 분석.

### 내 시스템 실측 — 그리고 꼭 짚어야 할 구분

> ⚠️ **혼동 주의(내가 처음에 헷갈린 지점)**: "서브에이전트를 쓰는 스킬" ≠ "`context: fork` 스킬".

- **내 시스템에 `context: fork` 프론트매터를 쓰는 스킬은 아직 없다.** 세 스킬 어디에도 `context`/`agent` 필드가 없다(직접 확인).
- 헷갈리기 쉬운 사례 — `chapter-deck`은 description에 "두 개의 병렬 서브에이전트로 동시에 생성"이라고 적혀 있다. 하지만 이건 **본문에서 Task 도구로 서브에이전트를 띄우는** 방식이지, **프론트매터 `context: fork`로 스킬 전체가 포크되는** 것과는 다른 메커니즘이다.
  - 차이의 핵심: `context: fork`는 **스킬 자체가 격리 컨텍스트에서 실행**(부모 히스토리 차단)된다. 반면 본문 Task 병렬화는 **부모 컨텍스트 안에서** 내가 추가로 일꾼을 부르는 것 — 부모 히스토리·다른 결과에 닿을 수 있다.
- **도입한다면 어디에**: `study-chapter`의 4단계(내 클로드 시스템 적용예시 실측)가 1순위 후보다. 그 단계는 `~/.claude/` 전반을 뒤지느라 토큰을 많이 먹고 부모 대화와 섞일 필요가 없다. `context: fork` + `agent: Explore` + `allowed-tools`(Read·Bash 정도)로 떼어내면, **탐색은 격리된 서브에이전트가 하고 정리본 작성 컨텍스트엔 결과만** 돌아온다 — 정확히 3-3이 말한 용법이다.

## 읽으며 생긴 질문과 내 결론

- **Q. `disable-model-invocation`과 `context: fork`는 같이 쓸 수 있나?** → 별개 축이다. 전자는 *호출 방식*(자동 차단), 후자는 *실행 위치*(격리). `study-chapter`처럼 사이드이펙트가 있으면서 무거운 탐색을 하는 스킬이라면 둘 다 켜는 게 자연스럽다(슬래시로만 부르되, 탐색은 포크해서).
- **Q. "팀 스킬 개발은 시기상조"라는 메모 — 정말?** → 적어도 이 레포는 반례다. `study-chapter`·`order-session`을 5명이 공유해 쓰니, "프로젝트 스킬을 git에 커밋해 팀이 공유"는 이미 작동 중인 패턴이다. 다만 트리거 충돌·버전 관리 비용이 커지는 *규모*에서 시기상조라는 뜻으로 읽었다.

**이번 주 발표용 핵심 개념 1개**: **`context: fork` — 스킬을 격리 실행 에이전트로 격상**.
임정이 "컨텍스트 윈도우는 공용자산(점진적 작동)"으로 3장의 *절약* 축을 가져갔다면, 나는 3-3의 *격상* 축을 맡는다. 둘이 합쳐지면 "공용자산을 아끼는 지식 모듈(3-1·3-2) → 그 모듈을 격리 실행 단위로 끌어올리기(3-3)"로 3장이 통째로 커버된다. 내 실측 포인트는 **"서브에이전트 쓰는 스킬 ≠ `context: fork` 스킬"** 이라는 구분과, 내 `study-chapter`를 어디서 포크하면 좋은지의 구체적 제안이다.
