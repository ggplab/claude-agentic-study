# 이정연 — 3장 정리: 클로드 스킬 (2026-06-21)

## 3-1. 클로드 스킬의 개념과 구조

- 컨텍스트 윈도우는 공용 자산이다: 유용한 지식과 패턴을 공유 가능한 형태로 캡슐화한다
- 스킬 저장소를 git에 커밋하면 팀원이 pull 받는 즉시 해당 지식에 접근
- 빠르게 온보딩

### 스킬의 유형
- 개인 스킬
- 프로젝트 스킬
- 플러그인 스킬

### 스킬 디렉터리 기본 구조
- SKILL.md 필수
- 나머지는 스킬의 복잡도에 따라 선택적으로 포함

### SKILL.md 파일 구조
- name: 스킬이 수행하는 행위를 나타내는 동명사
- description: 기능과 사용 트리거
- 마크다운 형식의 본문: 해당 스킬을 사용할 때 클로드가 따라야 할 지침, 유의사항, 예시 등
- 200줄 이내 유지: 하나의 스킬이 single responsibility principle을 따르도록

### 3단계 점진적 작동 방식
- 1단계 메타데이터 레벨 자동 로딩: name, description
- 2단계 지침 레벨: 요청이 특정 스킬의 description과 매칭되면 bash로 SKILL.md를 컨텍스트 윈도우에 추가
- 3단계 리소스와 코드 레벨
- 단계별로 토큰 소요 비용이 다르고 체계적으로 점진적으로 공개한다는 점이 신기했음

### 스킬 컨텍스트 비용 — 실전 최적화 전략
- description의 토큰 밀도를 극대화해 한정된 예산(100토큰) 내에서 최대한의 발견 가능성을 확보
- 클로드가 이미 알고 있는 지식은 SKILL.md 본문에서 제거. 범용 지식 같은 내용은 포함하지 않는다. 프로젝트 고유의 컨벤션, 팀 특화 패턴, 외부에 공개되지 않은 내부 API 명세 등이 SKILL.md에 필요한 정보
- 지원 파일의 구조를 도메인별로 분리. 단 지원 파일이 다른 지원 파일을 불러오지 않도록 주의 → 클로드가 파일을 부분적으로만 읽을 위험이 있다
- 스킬의 입출력 예시를 최적화한다. 가장 대표적이고 다양한 예시 1-2개만 SKILL.md에 포함하고 나머지 예시(엣지 케이스 등)는 EXAMPLES.md로 분리
- 스크립트 우선 전략: 실행 가능한 스크립트는 결과만 컨텍스트에 반영해 토큰 비용이 극적으로 낮음. 복잡한 검증 규칙, 데이터 변환 로직, 반복 처리 절차 등은 스크립트로 만들어 scripts/에 넣어둔다. SKILL.md에는 스크립트의 사용법만 써놓으면 된다
- 전체 컨텍스트 윈도우의 15% 이내를 스킬 토큰 비용으로 사용하는 것을 목표

### 도구 접근 권한 제한
- allowed-tools: 클로드가 사용하는 도구 제한

## 3-2. 스킬 사용과 개발

### 스킬 목록과 스킬 조합
- 복잡한 작업을 수행할 때는 여러 스킬을 조합하여 사용할 수 있음
- 자칫 유사한 기능의 스킬 여러 개가 동시에 활성화될 수 있기 때문에 각 스킬은 고유하고 구체적인 트리거 용어를 description에 지정해야 한다

### 커스텀 스킬 개발
- 1단계 목적 정의
- 2단계 SKILL.md 작성: 입출력 예시 세트를 다양하게 2개 이상 작성해 클로드가 패턴을 정확하게 인지하도록 돕는다
- 3단계 지원 파일 추가

### 스킬에서 환각 방지 지침 작성
- 불확실성 표현 허용 지침: 모르겠으면 모르겠다고 말할 수 있게 해주기
- 지식 범위 제한 지침: 참조할 수 있는 정보의 범위 제한
- 출처 명시 지침: 정보의 출처를 명시하도록 해서 신뢰성 확보

### 코드 실행 통합
- 스크립트는 클로드가 토큰을 소비하지 않고 결정론적으로 처리해야 하는 작업에 적합
- 스크립트의 오류 상황은 클로드에게 넘기지 않고 스크립트 안에서 직접 처리
  - 실행 권한
  - 종속성 설치 여부
  - 경로가 유닉스 스타일 슬래시인지 여부
- 스크립트를 포함한 스킬의 디렉터리

```
Skill-example/
- SKILL.md
- reference.md
- scripts/
    - format_data.py
    - validate_schema.py
    - requirements.txt
- templates/
    - data_template.json
```

### 테스트와 검증 방법
1. description이 구체적인지 확인
2. 스킬 디렉터리 경로 확인
3. YAML 유효성 검사
4. 디버그 모드 활용: claude --debug를 실행하여 스킬 로딩 문제 확인

## 3-3. 스킬 고급 사용과 최적화

- 동적 스킬 로딩 패턴
- 스킬 성능 최적화: 토큰 효율성과 응답 품질의 균형을 찾는 작업
- 스킬 버전 관리: SKILL.md 콘텐츠 내에 Version History 섹션을 포함하여 변경 사항을 문서화하는 것을 권장
- 팀 스킬 개발: 아직은 좀 시기상조의 느낌
- 디버깅 기법: claude --debug 명령으로 스킬 로딩 과정에서 발생하는 문제를 확인

### 재사용 가능한 스킬 패턴
- 특정 프로젝트나 기술 스택에 종속되지 않고 일반적인 패턴을 다룬다
- 다양한 상황에 적용할 수 있도록 description에 충분한 트리거 조건을 포함한다
- 명확한 체크리스트와 단계별 지침을 가져 다른 스킬과 조합하기 쉽다

### 패턴 종류
- 프로젝트 특화 패턴: 특정 프로젝트의 규칙과 관행
- 도메인 특화 패턴: 특정 기술 스택이나 프레임워크에 특화
- 범용 유틸리티 패턴: 가장 재사용성이 높은 형태. 코드 리뷰, 문서 생성, 테스트 작성, 리팩터링 등

### 스킬 문서화 유의 사항
- 스킬 자체도 문서지만 스킬에 대한 메타 문서화도 중요
- 스킬 문서화에 포함해야 할 핵심 요소: 스킬의 목적과 적용 범위, 실제 사용 예시, 다른 스킬과의 관계(조합 패턴), 알려진 제한 사항이나 주의 사항

### 스킬의 격리 실행: context: fork 메커니즘
- context: fork가 설정된 스킬은 부모 대화로부터 완전히 분리된 서브에이전트 컨텍스트에서 실행
- 스킬 본문이 서브에이전트를 구동하는 프롬프트가 되며 서브에이전트는 부모 대화의 히스토리에 접근할 수 없음
- 작업이 완료되면 결과만 부모 대화에 반환
- agent 필드는 context: fork가 설정된 경우에만 동작하며 서브에이전트의 동작 모드를 결정한다
  - Explore: 코드베이스 탐색
  - Plan: 컨텍스트를 수집하고 실행 계획 수립
  - general-purpose: 기본값으로 별도 제약 없음
- 사용하기 적합한 경우: 대규모 코드베이스 분석, 세션 격리가 필요한 워크플로, 도구 전문화 기반의 분석 작업
- context: fork와 allowed-tools를 조합하면 격리와 보안을 동시에 달성할 수 있음
- 스킬 시스템을 단순한 지식 모듈에서 독립 실행 가능한 에이전트 단위로 격상시키는 메커니즘
- 서브에이전트의 격리된 컨텍스트를 활용하면서도 스킬 고유의 자동 발견과 동적 활성화 장점을 그대로 유지할 수 있음
