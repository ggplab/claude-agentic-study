§1 스킬 유형·위치

어디에 두느냐가 범위를 결정

개인(글로벌) 스킬

~/.claude/skills/
  • 모든 프로젝트에서 공통 사용
  • 실측: cmux, team, shorts, codebase-memory-* 등 수십 개
  • 어느 프로젝트에서든 description 뜸

프로젝트 스킬

<프로젝트>/.claude/skills/
  • 해당 프로젝트에서만 활성
  • 실측: study-chapter, chapter-deck, order-session 3개
  • 스터디 저장소에서만 의미있는 스킬 → 프로젝트 쪽이 맞다
정정: projects/.claude/skills가 아니라 프로젝트 루트의 .claude/skills/ — projects/라는 고정 폴더가 있는 게 아님
§2 디렉토리 구조

1depth-flat 원칙

pdf-skill/
├── SKILL.md ← 필수. 대소문자 정확히
├── FORMS.md ← 양식 채우기 가이드
├── REFERENCE.md ← 상세 API 레퍼런스
└── scripts/
    └── fill_form.py
  • 헬퍼 파일은 1depth 이내 — 깊게 중첩하지 않는다(토큰·탐색 효율)
  • SKILL.md는 대소문자가 정확해야 인식됨
  • 실측: 내 프로젝트 스킬 3개 모두 SKILL.md 단일 파일(분량 작아 헬퍼 불필요)
§3 프론트매터

name · description이 전부

필드의미
name스킬 목록에 보이는 이름(생략 시 폴더명)
description무엇을+언제 사용. 클로드가 호출 여부를 판단하는 근거. when_to_use와 합쳐 1,536자 제한
disable-model-invocationtrue면 자동 로딩 금지 → /name 수동 호출만
argument-hint자동완성 힌트. 예: [챕터번호]
context: fork격리된 서브에이전트 컨텍스트에서 실행(§9)
실측 — disable-model-invocation: study-chapter true chapter-deck true order-session 없음(자동)
왜 다른가? — 정리·슬라이드 스킬은 시점을 잡아 호출, 순서조정은 자동 호출돼도 무해
§4 ★ 핵심 — 3단계 점진적 로딩

비용은 개수가 아니라 "열어본 횟수"

L1
메타데이터 선제 로딩
모든 스킬의 name + description만 미리 컨텍스트에 올린다
전 스킬
L2
SKILL.md 본문 로딩
지금 요청이 description과 매칭되면 그때 본문을 컨텍스트에 추가
매칭 시
L3
리소스 개별 로딩
REFERENCE.md, scripts/* 등 실제로 필요한 순간에 하나씩
필요 시
산 증거 — 이 세션에서 수십 개 스킬이 description 한 줄씩만 떠 있다가, /study-chapter 호출 시 그제서야 SKILL.md 67줄이 들어왔다
§4 확률 vs 결정론

지침은 확률적, hook은 결정론적

description · SKILL.md 지침

  • 읽고 따름 — 가끔 새는 경우 있음
  • 강제력 없음
  • 예: "적용예시는 실측, 추정 금지"

hook 게이트

🛡
  • 코드가 막음 — 100% 보장
  • 결정론적 차단
  • 실측: cbm-code-discovery-gate PreToolUse가 실제로 내 Read를 막음
결론 — 확실히 보장해야 하는 게이트는 확률(지침)에 맡기지 말고 hook으로 결정론적으로 건다
§5 description 설계

발견가능성↑ vs 과범용↓

  • 밀도↑ 발견가능성↑ — 핵심 use case를 앞에, when_to_use와 합쳐 1,536자 한도
  • 과범용 위험 — 너무 넓으면 불필요한 상황에도 튀어나옴. 행위·주체·시점·입력을 좁혀 써야
  • 직접 쓰기 어렵다면 skill-creator 스킬 활용

좁히기 성공 — order-session

"단원 리더가 일요일 오후에 참여자들이 올린 글을 보고 발표 순서를 조정한다 … 인자로 챕터 번호를 받는다."

과범용 교훈 — 메모리 기록

SKILL.md에 훅 우회 지침 넣었다가 git add 단계에서 classifier에 차단됨 → description/지침은 넓게 잡으려다 사고

§6 예시 전달 & EXAMPLES.md 격리

예시는 파일로 격리하면 토큰 효율

  • 클로드는 예시를 주면 지침을 더 잘 따른다
  • 예시까지 다 넣으면 SKILL.md가 비대 → 중간 흘려읽기로 오히려 놓침
  • 자세한 예시는 EXAMPLES.md로 격리(1depth-flat) → L3 단계에서 개별 로딩
실측 — 내 스킬 현황
  • 현재 3개 스킬 모두 예시 본문에 인라인(분량 작아 격리 불필요)
  • 도입 1순위: chapter-deck의 슬라이드 HTML 예시가 길어지면 EXAMPLES.md로 분리
§7 환각 방지

SKILL.md에 넣는 방지 지침 3종

1
불확실성 표현 허용
모르면 "모른다/불확실"이라고 말하게 허용
허용
2
지식 범위 제한
확인된 범위 밖은 추정 금지 — 파일을 읽고 확인된 사실만
제한
3
출처 명시
근거가 된 파일·사실을 밝히게. 없으면 "아직 없음 + 도입한다면 어디에"로
명시
실측 — study-chapter: "적용예시는 실측, 추정 금지 — 사용자의 실제 설정 파일을 읽고 확인된 사실만 쓴다" / order-session: "날짜 추정 금지(date 실행)"
§8 CLAUDE.md & @import

카파시 4원칙 + "사람은 방향, AI는 실행"

카파시(Karpathy) CLAUDE.md

  • Think Before Coding — 가정 명시·혼란 숨기지 말 것
  • Simplicity First — 요청 이상 만들지 말 것
  • Surgical Changes — 요청과 직접 연결되는 줄만 수정
  • Goal-Driven — 검증 가능한 성공 기준으로 루프

내 CLAUDE.md — 실측

  • "사람은 방향, AI는 실행" = Surgical+Goal
  • "When unsure, ask — don't guess" = Think Before Coding
  • "Never touch production config without explicit permission" = Simplicity First
  • @import는 아직 안 씀(grep 결과 없음)
도입한다면: "Rules (update when Claude makes mistakes)" 블록이 길어지면 @rules.md로 빼서 CLAUDE.md 본문을 가볍게 유지
§9 context: fork

격리의 이득과 중복 로딩 단점

fork 없음 (메인 컨텍스트)

컨텍스트 오염 가능

  • 스킬 실행이 메인 히스토리에 쌓임
  • 대량 Read → 메인 컨텍스트 팽창
+토큰↑

context: fork 설정 시

격리된 서브에이전트

  • 메인 컨텍스트를 더럽히지 않음
  • 단점: 메인이 읽은 파일을 fork가 또 읽어 중복 토큰
실측 — 내 스킬 어디에도 context: fork 없음. 대신 chapter-deck은 "2개 서브에이전트로 병렬 디스패치, 메인 맥락을 물려주지 않는다"로 동일 효과 달성
