## 1. 스킬의 역할 — 반복되는 작업을 ‘실행 가능한 지침’으로 만든다

스킬은 특정 작업에 필요한 지식, 절차, 참고 자료, 스크립트를 한 디렉터리에 묶은 단위다. 핵심 파일은 `SKILL.md`이며, 보통 YAML 프론트매터와 마크다운 지침 본문으로 구성한다.

- 프론트매터: 스킬 이름, 언제 쓸지(`description`), 사용할 도구 등 스킬의 메타데이터
- 본문: 작업 순서, 제약, 출력 형식, 확인 방법
- 지원 파일: 상세 참고문서, 템플릿, 실행 스크립트 등

좋은 스킬은 “무엇을 할 수 있는가”보다 **어떤 요청에서 호출되어야 하는가**를 분명하게 설명한다. 비슷한 기능의 스킬이 겹치지 않도록 `description`에는 `데이터 처리`처럼 넓은 표현보다 `CSV 파일 변환`, `Excel 스프레드시트 분석`처럼 구체적인 트리거를 넣는다.

## 2. 점진적 로딩 — 컨텍스트는 필요한 만큼만 쓴다

스킬은 모든 내용을 한 번에 읽는 구조가 아니라, 작업의 복잡도에 따라 정보를 나누어 읽는 구조가 효율적이다.

| 단계 | 로드하는 내용 | 목적 |
| --- | --- | --- |
| 1단계 | 이름·설명 등 메타데이터 | 어떤 스킬을 쓸지 판단 |
| 2단계 | `SKILL.md`의 핵심 지침 | 작업 절차와 결과 형식 이해 |
| 3단계 | 참고문서·템플릿·스크립트 | 실제 작업에 필요한 세부 정보 사용 |

이 구조의 요점은 사용하지 않는 정보가 대화 컨텍스트를 차지하지 않게 하는 것이다. 본문에는 핵심 원칙과 분기 기준을 두고, 자세한 API 설명·많은 예시·대용량 데이터는 별도 파일로 분리한다.

### 스킬 내용을 가볍게 만드는 기준

- AI가 이미 아는 일반 지식은 길게 반복하지 않는다.
- 프로젝트 고유 규칙, 내부 API, 팀의 명명 규칙처럼 **AI가 알 수 없는 정보**를 우선 기록한다.
- 예시는 대표적인 1~2개만 본문에 두고, 나머지는 별도 예시 파일로 분리한다.
- 정형 변환·검증처럼 규칙이 명확한 작업은 설명으로 길게 풀기보다 스크립트로 처리한다.
- 지원 파일은 한 단계 깊이에서 접근할 수 있게 구성해 참조 경로를 단순하게 유지한다.

## 3. 커스텀 스킬 설계 순서

스킬을 만들기 전에 ‘무엇을 자동화할지’와 ‘언제 실행할지’를 먼저 좁혀야 한다. 범위가 넓으면 호출 조건이 모호해지고, 결과도 일관되기 어렵다.

1. 작업 목적과 범위를 정한다.
2. 호출 문장과 구체적인 트리거 키워드를 정한다.
3. 필요한 도구와 권한을 최소 범위로 정한다.
4. `SKILL.md`에 목표, 절차, 입력·출력 형식, 실패 시 처리 방법을 적는다.
5. 필요한 경우 `reference.md`, `scripts/`, `templates/`를 추가한다.
6. 대표 요청과 경계 사례로 호출·결과를 검증한다.

### 최소 구조 예시

```text
my-skill/
├── SKILL.md          # 호출 조건과 핵심 작업 절차
├── reference.md      # 상세 규칙·예시
├── scripts/          # 결정론적으로 처리할 작업
└── templates/        # 결과물의 형식
```

## 4. 신뢰할 수 있는 스킬을 위한 안전장치

스킬의 지침은 결과의 신뢰도를 좌우한다. 특히 외부 문서, 라이브러리 버전, API 설정처럼 최신성이나 사실 확인이 필요한 정보에는 추측을 허용하지 않는 기준이 필요하다.

- 확인하지 못한 값은 단정하지 않고 `공식 문서에서 확인 필요`라고 명시한다.
- 프로젝트 분석 결과는 소스 코드, 의존성 파일, README처럼 확인 가능한 파일을 근거로 남긴다.
- 외부 기술 정보에는 공식 문서 URL, 설정 정보에는 파일 경로 등 출처를 함께 제시한다.
- 읽기·검색만 필요한 작업에는 읽기 전용 도구만 허용해 변경 위험을 줄인다.
- 스크립트의 입력·출력·오류 코드와 오류 메시지를 문서화한다.

## 5. 스크립트와 테스트

반복되고 결과가 명확한 작업은 모델의 서술보다 스크립트에 맡기는 편이 안정적이다. 예를 들어 데이터 형식 변환, 스키마 검증, 파일명 정규화, 보고서의 기계적 검사 등이 이에 해당한다.

스킬을 검증할 때는 다음을 확인한다.

- `description`이 실제 요청과 잘 매칭되는가?
- 스킬 파일의 위치와 YAML 형식이 올바른가?
- 필요한 도구만 사용하도록 제한됐는가?
- 정상 입력뿐 아니라 누락·잘못된 형식 같은 실패 입력도 다루는가?
- 스크립트를 독립 실행했을 때 입력·출력·오류 처리가 문서와 일치하는가?

## 6. 고급 활용: 분리, 재사용, 문서화

### 무거운 작업은 격리한다

대규모 코드베이스 탐색처럼 중간 결과가 많은 작업은 별도의 컨텍스트에서 수행하고, 최종 요약만 가져오는 방식이 적합하다. 반대로 사용자의 선호나 앞선 대화 맥락을 반드시 알아야 하는 짧은 작업은 분리하지 않는 편이 낫다.

### 재사용 범위로 스킬을 구분한다

| 유형 | 담는 내용 | 적합한 범위 |
| --- | --- | --- |
| 프로젝트 특화 | 아키텍처, 코딩 규칙, 팀 관행 | 특정 프로젝트 |
| 도메인 특화 | React, Django, FastAPI 등 기술별 패턴 | 해당 기술을 쓰는 프로젝트 |
| 범용 유틸리티 | 코드 리뷰, 문서 생성, 테스트 작성 | 여러 프로젝트 |

스킬을 잘 문서화하려면 목적·적용 범위·호출 예시·다른 스킬과의 관계·알려진 제한 사항을 남긴다. 팀에서 함께 쓴다면 스킬 목록을 따로 관리하는 것도 발견성과 유지보수에 도움이 된다.

## 산출물: 스킬 설계 체크리스트

아래 체크리스트는 새 스킬을 만들거나 기존 스킬을 정리할 때 바로 사용할 수 있다.

### A. 설계

- [ ] 이 스킬이 해결할 작업을 한 문장으로 말할 수 있다.
- [ ] 스킬이 호출될 구체적인 요청 표현을 2개 이상 정했다.
- [ ] 다른 스킬과 역할 또는 트리거가 겹치지 않는다.
- [ ] 개인·프로젝트·도메인·범용 중 재사용 범위를 정했다.

### B. `SKILL.md`

- [ ] `description`에 작업 대상과 호출 상황을 구체적으로 적었다.
- [ ] 입력값, 작업 순서, 기대 출력 형식이 있다.
- [ ] 불확실한 정보는 확인하도록 지시하고 추측을 금지했다.
- [ ] 필요한 도구와 권한만 허용했다.
- [ ] 실패·예외 상황의 처리 기준이 있다.

### C. 지원 파일과 품질

- [ ] 핵심 지침만 `SKILL.md`에 두고 상세 정보는 분리했다.
- [ ] 스크립트에는 입력·출력·오류 처리 방식이 문서화돼 있다.
- [ ] 대표 요청, 복잡한 요청, 실패 입력으로 테스트했다.
- [ ] 결과에 출처 또는 확인 근거를 남겨야 하는 경우 그 형식을 정했다.

| 스터디 정리 | “오늘 스터디 내용을 정리해줘” | 핵심 개념·질문·다음 할 일 Markdown |
| 사진 기반 노트 | “이 책 사진을 정리해줘” | 사진 범위별 요약·개인 인사이트·체크리스트 |
| 코드 리뷰 | “이 변경사항을 검토해줘” | 필수 수정·권장 수정·확인 항목으로 구분한 리뷰 |

**이번 장의 결론:** 스킬은 지식을 많이 넣는 문서가 아니라, 필요한 정보를 적절한 시점에 꺼내고 안정적으로 실행하도록 설계하는 작업 인터페이스다.
