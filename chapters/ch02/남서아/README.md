# CHAPTER 02 정리 — 남서아

> 2장 "워크플로와 설정"을 읽으며 든 질문은 하나였다. "설정을 내가 잘 쓰는 데 그치지 않고, 비개발자 동료가 안전하게 쓰게 어떻게 넘기지?" 전산팀 없는 회사에서 혼자 사내 AI 도구를 맡다 보니 이게 가장 현실적인 문제였다.

---

## 1. settings.json과 권한 — 하네스의 본체

클로드의 동작·권한·모델·훅·MCP를 미리 정해두는 파일이다. 계층은 전역 → 프로젝트 → 로컬 순으로 아래가 우선한다. 권한은 `permissions.allow / ask / deny`로 "무엇을 자동 통과시키고 무엇을 멈춰 물을지"를 정한다.

**내 시스템 실측:** 내가 만든 모니터링 도구의 `.claude/settings.json`은 `allow`에 `Bash(python3 *)`, `Read/Write/Edit/WebFetch` 등을 다 열어두고 `deny`는 비어 있었다. 프로토타입을 빨리 굴리려고 "일단 다 열어둔" 상태. 이정연이 말한 "어디에 게이트를 둘 것인가"가 내 파일엔 아직 없다는 걸 실측으로 확인했다. 배포용으로는 개발자 실경로를 뺀 `settings.dist.json`을 따로 두고 포장 시 치환하게 해뒀다.

---

## 2. managed 설정 — 관리자가 강제하는 최상위 계층

조직 관리자가 사용자가 바꿀 수 없게 강제하는 시스템 레벨 파일이다. 위 계층 전부보다 우선한다. 경로는 macOS `/Library/Application Support/ClaudeCode/managed-settings.json`, Linux `/etc/claude-code/managed-settings.json`, Windows `C:\ProgramData\ClaudeCode\managed-settings.json`.

Teams/Enterprise 전용이 아니다. 시스템 경로에 파일을 둘 수 있는 관리자 권한(루트/MDM)이 있으면 쓸 수 있다. 개인 한 대에서도 실험은 되지만, 실익은 MDM으로 여러 대에 배포할 때 생긴다.

**내게 의미:** "반복은 열고 중대한 건 잠근다"를 동료가 못 풀게 하려면 개인 권한 게이트(이정연 주제)만으로는 부족하다. 이 강제 계층이 필요하다.

---

## 3. 스킬 로드 구조

스킬은 `SKILL.md`(frontmatter + 본문) + 지원 파일로 구성된다. 클로드는 frontmatter의 `description`을 보고 맥락에 맞을 때만 본문을 로드한다. 스킬이 많아도 평소엔 요약만 컨텍스트에 있고 쓸 때만 펼쳐진다.

**내 시스템 실측:** 내 도구엔 스킬 2개가 있다. `article-extractor`(본문 추출 → 사이트별 셀렉터 → 폴백)와 `briefing-formatter`(HTML 렌더). 둘 다 판단이 거의 없는 결정론적 작업이라 스킬로 묶어뒀는데, 이 절 개념과 잘 맞아떨어졌다.

---

## 4. 서브에이전트 vs 에이전트 팀

**서브에이전트:** 메인 에이전트가 띄우는 격리된 1회성 워커. 독립 컨텍스트에서 일하고 결과 하나를 돌려준다. 서로 직접 대화 못 하고 공유가 필요하면 파일을 거친다. → "던지고 결과 받기(fan-out)". 예: 기사 50건을 각각 병렬 요약.

**에이전트 팀:** 이름이 붙은 여러 에이전트가 메시지로 협업하는 구조(리드-워커). → "대화하며 같이 만들기(coordination)". 예: 설계자 → 구현자 → 검수자가 한 기능을 주고받으며 완성.

**내 도구는 사실상 전자다.** 단계별로 던지고 받는 파이프라인이라 팀보다 서브에이전트/순차 호출 모델에 가깝다.

---

## 5. MCP 컨텍스트 절약과 managed-mcp

MCP 서버를 연결하면 그 서버의 도구 정의(이름·설명·스키마)가 컨텍스트에 상주한다. 서버가 많을수록 매 턴 토큰을 깐다. 안 쓰는 서버는 꺼두면 절약된다. `/mcp`로 토글하거나 설정에서 필요한 것만 켜면 된다. 최신 클로드 코드는 필요할 때만 스키마를 불러오는 지연 로딩도 쓴다.

조직 단위로 MCP 서버를 강제하거나 차단하는 건 managed 설정 계층에서 처리한다. 별도 파일이 아니라 `managed-settings.json` 안의 키로 제어하는 구조다.

**내게 의미:** 비기술자 배포 시 "필요한 MCP만, 나머지는 끄기"로 토큰과 혼란을 줄인다.

---

## 6. Hooks — 결정론적 실행

훅은 특정 이벤트가 발생하면 무조건 셸 명령을 실행한다. 모델이 "부를지 말지" 판단하는 스킬·서브에이전트와 다르게, 발화가 보장된다. 이벤트 종류: `PreToolUse`, `PostToolUse`, `UserPromptSubmit`, `Notification`, `Stop`, `SubagentStop`, `PreCompact`, `SessionStart`, `SessionEnd`.

**내 시스템 실측:** 내 도구의 결정론은 전부 `run-pre.sh`(fetch → extract → classify → aggregate, LLM 없음) 같은 쉘 파이프라인으로 구현돼 있고 `hooks` 키는 0건이었다. 이 방식이 틀린 건 아니지만, "매 편집 후 포맷", "위험한 Bash를 실행 전 차단", "세션 시작 시 맥락 주입"처럼 라이프사이클 지점에 끼워야 하는 일은 직선 스크립트로 못 잡는다. 그게 훅의 자리였다.

다음 단계로 품질 게이트·실행 전 검증을 훅으로 옮겨볼 예정이다.

---

## 7. opusplan

계획은 Opus, 구현은 Sonnet으로 자동 분담하는 하이브리드 모드다. `/model opusplan`으로 켜거나 `settings.json`에 `"model": "opusplan"`으로 둔다.

**내 시스템 실측:** 인사이트 합성 에이전트는 Sonnet, 월간 맥락 갱신 에이전트는 Haiku로 frontmatter에 박아뒀다. 에이전트 단위 model override로 이미 작업별 모델 분리를 하고 있었던 셈이다. opusplan은 여기에 "계획 vs 구현" 축을 하나 더 얹는 것.

---

## 8. 직접 적용 — 모니터링 도구를 배포 가능한 형태로

지금 만드는 PR·뉴스레터 모니터링 도구를 이 장 기준으로 점검했다.

- 결정론(수집·추출·분류·렌더·품질게이트)은 전부 Python/bash로 고정하고, LLM은 인사이트 합성과 보도 톤 판정 두 곳에만 개입한다. 훅으로 옮기면 더 네이티브해질 영역이다.
- 비기술자 친화 설계(슬래시 커맨드, 자연어 운영, 한글 사용설명서)는 갖춰져 있어 사내 워크스페이스 배포는 거의 가능하다.
- 다만 `.claude-plugin/plugin.json` 매니페스트가 없어서, 엄밀히는 "폴더 열기로 쓰는 워크스페이스"지 클로드 플러그인이 아니다. 플러그인으로 가려면 매니페스트 신설이 남았다.

이 장 덕분에 "배포 가능한 패키지가 되려면 무엇이 비었는지"가 체크리스트로 보였다.

---

## 🎤 발표용 핵심 개념

**"설정을 '나만 잘 쓰는 법'이 아니라 '남에게 안전하게 넘기는 법'으로 — settings.json 계층 + managed 강제 + 플러그인 패키징"**

임정은 도구 5가지의 트리거 분류, 이정연은 개인 권한 게이트, 최지환은 TDD를 골랐다. 이정연과 같은 settings.json을 다루지만 레이어가 다르다. 정연은 "내가 어디에 게이트를 둘까"(개인 워크플로), 나는 "관리자가 조직에 무엇을 강제·배포할까"(배포 레이어)다. 실제로 도구를 만들며 부딪힌 문제라 책 요약이 아니라 내 작업 이야기로 발표할 수 있다.
