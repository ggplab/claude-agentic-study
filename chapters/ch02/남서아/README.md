# CHAPTER 02 정리 — 남서아

> 2장 "워크플로와 설정"을 읽고 정리한 개인 노트. 절별로 **개념정리 + 내 시스템에서 실제로 확인한 적용예시**를 붙였다. 마지막에 발표용 핵심 개념 1개와 고른 이유.

내가 이 장에서 던진 질문은 하나였다. "설정을 내가 잘 쓰는 데 그치지 않고, 비개발자 동료가 안전하게 쓰게 어떻게 넘기지?" 전산팀이 없는 회사에서 혼자 사내 AI 도구를 맡다 보니, 제어 실패·보안 사고를 막는 하네스(settings.json)가 남 손에 넘어가도 버티는 형태여야 했기 때문이다.

> 공식문서 출처는 docs.claude.com/en/docs/claude-code 기준. 일부 정확한 파일 경로·스키마는 직접 환경에서 확인하는 걸 권장.

---

## 1. settings.json과 권한 — '하네스'의 본체

- **개념**: 클로드의 동작·권한·모델·훅·MCP를 미리 정해두는 설정 파일. 계층은 아래로 갈수록 우선한다: ① 전역 `~/.claude/settings.json` → ② 프로젝트 `.claude/settings.json` → ③ 로컬 `.claude/settings.local.json`. 권한은 `permissions.allow` / `ask` / `deny`로 "무엇을 자동 통과시키고 무엇을 멈춰 물을지"를 설계한다.
- **내 시스템 실측**: 내가 만든 모니터링 도구의 `.claude/settings.json`은 `permissions.allow`에 `Bash(python3 *)`, `Bash(./scripts/*)`, `Read/Write/Edit/WebFetch` 등을 광범위하게 열어두고 `deny`는 비어 있었다. 프로토타입을 빨리 돌리려고 "일단 다 열어둔" 상태였던 셈이다. 이정연이 말한 "어디에 게이트를 둘 것인가"가 내 파일에선 아직 설계되지 않았다는 걸 실측으로 확인했다. 배포용으로는 개발자 실경로가 박힌 `env`를 뺀 `settings.dist.json`을 따로 둬서 포장 시 치환하게 해놨다.

## 2. managed 설정(managed-settings.json) — 관리자가 강제하는 최상위 계층

- **개념**: 조직(관리자)이 사용자가 **바꿀 수 없게** 강제하는 시스템 레벨 정책 파일. 위 1~3계층보다 우선해서 덮어쓴다. 경로는 환경별로 macOS `/Library/Application Support/ClaudeCode/managed-settings.json`, Linux `/etc/claude-code/managed-settings.json`, Windows `C:\ProgramData\ClaudeCode\managed-settings.json`.
- **읽으며 생긴 질문과 내 결론**: "Teams/Enterprise를 못 쓰는 환경이면 managed 설정도 못 쓰나?" → 결론은 "구독 플랜 자체가 잠그는 기능은 아니다"이다. 이건 정해진 시스템 경로에 파일을 둘 수 있는 **관리자 권한(루트/MDM)**으로 동작하는 파일 기반 메커니즘이다. 즉 개인 한 대에서 만들어 실험은 되지만, 실익은 MDM으로 여러 대에 배포할 때 생긴다. (조직 차원 배포·강제는 보통 Enterprise 운영과 함께 가는 게 맞다. 정확한 플랜별 제약은 공식문서로 최종 확인.)
- **내게 의미**: 1인 전산팀이 "반복 작업은 열고 중대한 건 잠그는" 걸 동료가 못 풀게 하려면, 개인 권한 게이트(이정연 주제)만으로는 부족하고 이 강제 계층이 답이 된다.

## 3. 스킬 로드 구조

- **개념**: 각 스킬은 `SKILL.md`(frontmatter + 본문) + 스크립트·템플릿·참조문서 같은 지원 파일로 구성된다. 클로드는 frontmatter의 description을 보고 **요청 맥락에 맞을 때만 본문을 로드**한다(점진적 공개). 그래서 스킬이 많아도 평소엔 요약만 컨텍스트에 있고, 쓸 때만 펼쳐진다.
- **내 시스템 실측**: 내 도구엔 스킬 2개가 있다 — `article-extractor`(본문 추출: 라이브러리 추출 → 사이트별 셀렉터 → 외부 추출 폴백), `briefing-formatter`(HTML 렌더). 둘 다 `SKILL.md` + 파이썬 스크립트로 "판단이 거의 없는 결정론적 작업"을 스킬로 묶어둔 형태였다. 스킬을 "모델이 알아서 부르는 단위"로 쓴 거라, 이 절 개념과 잘 맞아떨어졌다.

## 4. 서브에이전트 vs 에이전트 팀 — 헷갈렸던 부분 정리

- **서브에이전트(subagent)**: 메인 에이전트가 띄우는 **격리된 1회성 워커**. 독립 컨텍스트에서 일하고 결과 하나를 메인에 돌려준다. 서로 직접 대화하지 못해서, 공유가 필요하면 공통 메모 파일을 거친다. → **독립적이고 병렬 가능한 작업, 컨텍스트 절약, 전문 역할 분담**에 적합. 예: "이 기사 50건을 각각 요약".
- **에이전트 팀(team)**: 이름이 붙은 여러 에이전트가 **메시지로 서로 협업**하는 구조(리드-워커). 지속적으로 오가며 조율한다. → **상호 의존·핑퐁이 필요한 작업**에 적합. 예: "설계자 → 구현자 → 검수자가 한 기능을 주고받으며 완성".
- **한 줄 차이**: 서브에이전트는 "던지고 결과 받기(fan-out)", 팀은 "대화하며 같이 만들기(coordination)". 내 모니터링 도구는 사실상 전자다 — 단계별로 던지고 받는 파이프라인이라, 팀보다 서브에이전트/순차 호출 모델에 가깝다.

## 5. managed-mcp.json과 MCP 컨텍스트 절약

- **MCP가 컨텍스트를 먹는 이유**: MCP 서버를 연결하면 그 서버의 **도구 정의(이름·설명·스키마)가 컨텍스트에 상주**한다. 서버·도구가 많을수록 매 턴 토큰을 깐다. 그래서 안 쓰는 서버를 꺼두면 토큰이 절약된다. 끄는 법은 `/mcp`로 토글하거나, 설정에서 프로젝트 MCP 자동 활성화를 끄고(`enableAllProjectMcpServers` 류) 필요한 것만 켜는 방식. (최신 클로드 코드는 도구 스키마를 평소엔 안 싣고 필요할 때만 불러오는 지연 로딩도 쓴다.)
- **managed-mcp.json(엔터프라이즈)**: 조직이 MCP 서버를 **중앙에서 강제·차단**하는 관리형 설정. managed 설정 계층에서 허용/차단 목록으로 통제한다. (정확한 파일명·스키마는 환경/버전마다 다를 수 있어 공식문서로 확인 필요 — 이 항목은 추가 검증 중.)
- **내게 의미**: 비기술자에게 배포할 땐 "필요한 MCP만, 나머지는 꺼서" 토큰·혼란을 줄이는 게 운영부담을 낮춘다.

## 6. Hooks — 결정론적 실행을 '네이티브하게'

- **개념**: 훅은 특정 **이벤트가 나면 무조건 셸 명령을 실행**한다. 모델이 "부를지 말지" 판단하는 스킬·서브에이전트와 달리, 발화가 보장돼서 결정론적이다. 이벤트 종류: `PreToolUse`, `PostToolUse`, `UserPromptSubmit`, `Notification`, `Stop`, `SubagentStop`, `PreCompact`, `SessionStart`, `SessionEnd`. settings.json의 `hooks` 키에 이벤트별 matcher와 명령을 적는다.
- **읽으며 생긴 질문과 내 결론**: "내가 쉘스크립트로 전체 커맨드를 묶어 직선 플로우로 돌린 게 네이티브하지 않았던 듯?" → 맞다. 내 도구의 결정론은 전부 `run-pre.sh`(fetch → extract → classify → aggregate, LLM 없음) 같은 **쉘스크립트 직선 파이프라인**으로 구현돼 있고 `hooks` 키는 한 곳도 없었다(전체 검색 0건). 이 방식이 틀린 건 아니지만, "매 편집 후 포맷", "위험한 Bash를 실행 전 차단", "세션 시작 시 맥락 주입"처럼 **라이프사이클 지점에 자동으로 끼우는 일**은 직선 스크립트로는 안 잡힌다. 그건 훅의 자리였다.
- **할 일**: 품질 게이트·실행 전 검증 같은 "무조건 돌아야 하는" 부분을 훅으로 옮겨본다.

## 7. opusplan과 모델별 설정

- **opusplan**: 공식 하이브리드 모드. **계획은 Opus, 구현은 Sonnet**으로 자동 분담한다. `/model opusplan`으로 켜거나 settings.json `"model": "opusplan"`으로 둔다. 어려운 판단(계획)에만 비싼 모델을 쓰고 반복 구현은 싼 모델로 내려 비용·품질 균형을 맞춘다.
- **모델별 설정 실측**: "작업별로 모델을 나눈다"는 발상은 내 도구가 이미 부분적으로 하고 있었다 — 인사이트 합성 에이전트는 Sonnet, 월간 맥락 갱신 에이전트는 Haiku로 frontmatter에 박아뒀다. 즉 권한별 모델 지정 같은 직접 기능 대신, **에이전트/스킬 단위 model override**로 사실상 작업별 모델 분리를 하고 있었던 셈. opusplan은 여기에 "계획 vs 구현" 축을 하나 더 얹는 것.

## 8. 직접 적용 중인 것 — 모니터링 도구를 '배포 가능한 형태'로

- 지금 만들고 있는 PR·뉴스레터 모니터링 도구를 첫 배포 대상으로 보고 있다. 구조를 이 장 기준으로 점검해보니:
  - 결정론(수집·추출·분류·렌더·출처조인·품질게이트)은 전부 Python/bash로 고정하고, LLM은 **판단 두 곳**(인사이트 합성, 보도 톤 판정)에만 개입하게 분리돼 있었다. → 6절 훅으로 옮기면 더 네이티브해질 영역.
  - 비기술자 친화 설계(슬래시 커맨드, 자연어 운영, 경로 독립, 가상환경 자동 구축, 한글 사용설명서)는 갖춰져 있어 "사내 워크스페이스 배포"는 거의 가능한 수준.
  - 다만 **`.claude-plugin/plugin.json` 매니페스트가 없어서**, 엄밀히는 "클로드 플러그인"이 아니라 ".claude 폴더를 zip으로 묶어 '폴더 열기'로 쓰는 워크스페이스"였다. 진짜 플러그인/마켓 배포로 가려면 매니페스트 신설 + 회사 도메인 지식의 템플릿화가 남았다.
- **요지**: 이 장 덕분에 "내 도구가 배포 가능한 패키지가 되려면 무엇이 비었는지"가 체크리스트로 보였다. 설정은 튜닝이 아니라 인수인계 문서였다.

## 🎤 발표용 핵심 개념 1개

**"설정을 '나만 잘 쓰는 법'이 아니라 '남에게 안전하게 넘기는 법'으로 — settings.json 계층 + managed 강제 + 플러그인 패키징"**

- **고른 이유**: 다른 분들과 안 겹친다. 임정은 도구 5가지의 트리거 분류, 이정연은 개인 권한 게이트 설계, 최지환은 TDD를 골랐다. 이정연과 같은 settings.json을 보지만 **레이어가 다르다** — 정연은 "내가 어디에 게이트를 둘까"(개인 워크플로), 나는 "관리자가 조직에 무엇을 강제·배포할까"(거버넌스·배포). 그리고 이건 내가 실제로 도구를 만들며 부딪힌 문제라, 책 요약이 아니라 내 작업 이야기로 발표할 수 있다.
