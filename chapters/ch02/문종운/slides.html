피드백 루프 철학

검증 견고화 vs 선택지 확장

클로드 코드 방식

  • 권한·훅·TDD로 검증을 견고하게
  • 실행 전 확인 단계 강제
  • 잘못된 방향 조기 차단
  • 안전성 ↑, 속도는 다소 낮음

Fable 방식 (실험적)

  • 제약 줄이고 선택지를 열어 둠
  • 더 넓은 탐색·실험 가능
  • 컨텍스트 로트 부분 해소
  • 자율도 ↑, 검증 책임 이동
※ 미확인 / 지원 중단 가능성 — Fable 모델은 스터디 중 "지원 중단" 이슈가 제기됨. 실험 환경에서도 확인 불가 상태. 아카이브 기준으로만 기록.

git worktree 실전

atomic 단위로 충돌 최소화

  • git worktree add ../wt-feat-a feat/a — 브랜치별 독립 작업 디렉터리 생성
  • 에이전트 A·B가 서로 다른 worktree에서 동시 작업 — 파일 충돌 원천 차단
  • 작업 단위를 작게 분할(atomic)할수록 merge 충돌 최소화
  • 완료 후 PR / merge 시점에 일괄 통합 — 통합 전까지 완전 격리
# worktree 생성 후 에이전트 실행 예시
git worktree add ../wt-a feat/auth
git worktree add ../wt-b feat/api
# 각 디렉터리에서 독립적으로 claude -p 실행
설정 파일

settings.json 개요

클로드 코드의 권한·동작을 선언적으로 제어

①permissions

  • deny — 절대 실행 금지
  • ask — 실행 전 확인
  • allow — 자동 허용

②hooks

  • 특정 이벤트에 스크립트 자동 실행
  • 예: 커밋 전 lint, 저장 후 테스트
  • 검증 견고화의 핵심 수단
{
  "permissions": {
    "deny": ["Bash(rm -rf:*)", "Bash(git push --f:*)"],
    "allow": ["Read", "Bash(npm test:*)"]
  }
}
permissions

deny / ask / allow — 색으로 한눈에

"이것이 검증을 견고하게 하는 방식" — 위험 명령을 선언적으로 차단

🚫 deny

절대 실행 금지 — 묻지도 않고 차단

Bash(rm -rf:*)
Bash(git push --f:*)
Bash(drop table:*)
Bash(curl * | sh)
❓ ask

실행 전 사용자 확인 — 기본값

Bash(git push:*)
Bash(npm publish:*)
Write(*)
Edit(*)
✅ allow

자동 허용 — 안전하다고 확인된 것만

Read(*)
Bash(npm test:*)
Bash(git status:*)
Bash(git diff:*)
IAM · 권한 관리

미설정 시 클로드는 모든 권한을 갖는다

  • 기본 상태의 클로드 코드는 파일·셸·네트워크 모두 접근 가능
  • 연결된 DB MCP가 있다면? — 테이블 전체 DROP 가능
  • 중요 저장소·서비스 접근은 deny로 사전에 차단
실제 사례 — 스터디 중 제기된 시나리오: MCP DB 연동 + 권한 미설정 → 클로드가 불필요한 정리 작업 수행 중 테이블 전체 삭제. permissions 설계는 선택이 아닌 필수.
커밋 추적성

Co-Authored-By & git 검증 에이전트

  • 클로드가 생성한 커밋에 Co-Authored-By: Claude 태그 자동 삽입
  • git 전용 서브에이전트를 두어 커밋 전 검증(lint · test · diff 확인)
  • AI 작성 코드와 사람 작성 코드를 명확히 추적 가능
  • 원하지 않는다면 includeCoAuthoredBy: false로 비활성화
# git commit message 예시
feat: add auth module

Co-authored-by: Claude <noreply@anthropic.com>

.claude/settings.json — 프로젝트
팀원과 공유·발전. 글로벌 위에 덮어씀. git에 커밋.

3
.claude/settings.local.json — 로컬
gitignore 대상. 개인 실험·임시 허용. 최종 우선.
최종 우선
  • 프로젝트 settings.json을 팀원이 함께 관리 → 공동 품질 게이트
비대화형 실행

claude -p — 자동화 파이프라인

  • claude -p "..." — 단일 프롬프트를 비대화형으로 실행, CI/CD에 적합
  • 권한에 막히면 응답 없이 무한 대기 — 자동화 환경에선 반드시 allow 설정 필요
  • --dangerously-skip-permissions — 모든 권한 확인 스킵
  • SDK(@anthropic-ai/claude-code)로 프로그래밍 방식 호출도 가능
⚠ 주의 — --dangerously-skip-permissions는 이름 그대로 위험. 신뢰된·격리된 환경(컨테이너, 전용 VM 등)에서만 사용할 것.
