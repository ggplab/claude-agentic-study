## 게시 본문

클로드 코드, 어떻게 하면 잘 활용할 수 있을까요? 저의 답은 세 가지로 분류하는 것입니다.

반드시 일어나야 하는 일이면 훅, 모델이 맥락을 보고 판단하면 좋은 일이면 스킬과 서브에이전트, 외부 연결이나 묶음 배포면 MCP와 플러그인입니다.

클로드 코드를 단순한 코딩 보조가 아니라 매일 쓰는 작업 시스템으로 운영하다 보면, 확장 요소가 다섯 가지라는 걸 알게 됩니다. 스킬, 서브에이전트, MCP 서버, 훅, 플러그인. 처음에는 다섯 개를 따로 쓰면서 언제 뭘 써야 하는지 헷갈렸습니다. 그런데 쓰다 보니 결국 하나의 질문으로 모이더군요. "이 일을 모델의 판단에 맡길 것인가, 아니면 기계적으로 강제할 것인가." 이 질문을 기준으로 다섯 가지를 정리해봅니다.

1. 스킬
한 줄 설명: 스킬은 클로드가 할 수 있는 일을 확장합니다. SKILL.md 파일에 지시를 적어두면 클로드가 자신의 도구함에 추가하고, 관련이 있을 때 꺼내 씁니다.
트리거: /스킬명 슬래시로 직접 부르거나(예: /commit), 설명이 맞으면 모델이 알아서 호출합니다.
내 활용: 30여 개를 쓰고 있습니다. 새 업무 기록, 자동화 실행 전 인증 점검 같은 반복 절차가 여기 들어갑니다. 핵심은 기능 설명이 아니라 "언제 나를 불러야 하는가"를 적어두는 것입니다. 다만 깃 푸시처럼 되돌리기 어려운 작업은 자동 호출을 꺼두고 슬래시로만 부르게 막아둡니다.

2. 서브에이전트
한 줄 설명: 서브에이전트는 특정 유형의 작업을 처리하는 전문화된 AI 어시스턴트입니다. 각자 별도의 컨텍스트 창과 시스템 프롬프트, 제한된 도구 권한을 가집니다.
트리거: "보안 점검 에이전트로 검토해줘"처럼 이름을 직접 부르거나, 설명에 맞는 작업이 오면 모델이 알아서 위임합니다.
내 활용: 작업을 별도의 대리인에게 맡겨 본 대화의 맥락을 깨끗하게 유지합니다. 여기서도 "무엇을 잘한다"가 아니라 "언제 위임할지" 조건을 적어야 제대로 선택됩니다. 저는 보안 사고 대응 에이전트의 설명에 의심스러운 키워드들을 직접 적어두어, 그런 상황이 감지되면 모델이 먼저 그 에이전트에게 넘기도록 설계했습니다.

3. MCP 서버
한 줄 설명: 클로드 코드는 MCP(모델 컨텍스트 프로토콜)라는 오픈소스 표준을 통해 수백 개의 외부 도구와 데이터 소스에 연결할 수 있습니다.
트리거: 슬래시 명령이 아니라, 연결해두면 "노션에서 이 문서 찾아줘"처럼 자연어로 요청할 때 모델이 그 도구를 사용합니다. 서버 연결 상태는 /mcp로 관리합니다.
내 활용: 노션, 브라우저 자동화, 그리고 홈서버 맥미니에서 돌아가는 또 하나의 AI 에이전트까지 도구로 붙여 두었습니다. 표준만 맞으면 꼭 거창한 서비스 API가 아니어도 무엇이든 도구가 된다는 점이 인상적이었습니다.

4. 훅
한 줄 설명: 훅은 클로드 코드 실행 주기의 특정 시점에 자동으로 실행되는, 사용자가 정의한 셸 명령입니다.
트리거: 명령어로 부르는 게 아닙니다. settings.json에 이벤트(예: 파일 수정 후, 명령 실행 직전, 세션 종료 시)로 등록하면 그 시점이 올 때마다 발화합니다.
내 활용: 확률적으로 움직이는 AI 위에서 무언가를 반드시 실행시킨다는 점이 특징이고, 저는 이게 에이전틱 도구의 핵심이라고 생각합니다. 파일을 고치면 타입 검사가 돌고, 외부 명령을 쓰기 직전 인증 상태를 점검하고, 세션이 끝나면 업무일지가 자동으로 기록됩니다. 모델이 깜빡해도 100퍼센트 실행된다는 것이 스킬과의 결정적 차이입니다.

5. 플러그인
한 줄 설명: 플러그인은 슬래시 명령, 에이전트, MCP 서버, 훅을 한 번의 명령으로 설치하는 맞춤 묶음입니다.
트리거: /plugin으로 설치하고 활성화합니다(예: /plugin marketplace add, /plugin install).
내 활용: 새로운 사용법이라기보다 앞의 네 가지를 담아 나르는 상자에 가깝습니다. 남이 만든 세트를 한 번에 설치하고, 안 쓰면 묶음째로 끄면 됩니다.

가장 마음에 드는 순간은 훅과 스킬이 섞일 때입니다. 제 시스템에는 사업 관련 키워드가 보이면 자동으로 작동하는 훅이 하나 있는데, 이 훅은 "사업 얘기 같다"는 1차 감지까지만 합니다. "정말 의사결정이 필요한 사안인가"라는 최종 판단은 모델에게 넘기죠. 판단이 서면, YC의 개리 탄이 진행하는 오피스아워 형식을 본떠 만든 사업 멘토링 스킬(yc-office-hours)을 불러 강제 질문으로 안건을 검증하게 했습니다. 이 스킬은 신규 아이디어나 수주 판단 같은 결정을 강제 질문으로 따져보고 대안 두세 개와 다음 행동 과제를 뽑아줍니다. 기계적인 강제와 유연한 판단이 한 흐름 안에 들어간 셈입니다.

결국 클로드 코드를 잘 쓰는 일은 도구를 더 많이 늘리는 게 아니라, 무엇을 모델에게 맡기고 무엇을 기계적으로 강제할지 선을 긋는 데서 시작한다고 생각합니다. 다섯 가지를 그 기준으로 다시 보면, 어디에 무엇을 둘지가 한결 또렷해집니다.

이번 글은 "클로드 코드로 시작하는 실전 에이전틱 코딩" 스터디의 일환으로 작성했습니다.
