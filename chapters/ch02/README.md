### 임정 — 도구 5가지를 '트리거 3종'으로 (스킬·서브에이전트·MCP·훅·플러그인)
- **내용 리마인드**: 클로드 코드의 확장 5요소는 "어떻게 호출되는가"로 보면 3가지 트리거로 환원된다. ① 모델이 설명을 읽고 자율 호출(스킬·서브에이전트), ② 이벤트에 무조건 발화하는 결정론(훅), ③ 둘을 담아 배포하는 컨테이너(MCP·플러그인). MCP·플러그인은 자체 트리거가 아니라 ①·②를 포장하는 단위.
- **나에게 어떤 의미였나**: 5개를 따로 외우다가 "이 일을 모델 판단에 맡길까, 기계적으로 강제할까"라는 한 질문으로 정리됐다. 내 시스템 실측(훅 6이벤트, 서브에이전트 5개, MCP 3개)을 이 프레임에 대입했고, 사업 키워드를 감지하는 훅(②)이 yc-office-hours 스킬(①)을 부르는 하이브리드가 이 축을 한 사례로 보여준다. (상세: [임정/README.md](임정/README.md) 4절 · IAM 발표안 아카이브: [임정/archive-iam-발표안.md](임정/archive-iam-발표안.md))
- **LinkedIn**: 게시 본문 초안 [임정/linkedin-5tools.md](임정/linkedin-5tools.md) (트리거 3분류로 본 도구 5가지) · 발행 후 링크 기입 예정

### 이정연 — 개발 프로세스 자동화를 위한 '권한 위임 시스템' 설계 (allow/ask/deny 게이트 디자인)
- **내용 리마인드**: 권한 설정의 진짜 쟁점은 "yes를 누를까 말까"가 아니라, 무엇을 자동 통과(allow)시키고 무엇을 반드시 멈춰 확인(ask/deny)하게 둘지 미리 설계하는 일이다. 전부 자동화하면 통제를 잃고, 매번 확인하면 흐름이 끊긴다. 좋은 권한 위임은 그 사이에 게이트를 의도적으로 배치하는 디자인 문제다.
- **나에게 어떤 의미였나**: 내 분야인 로보틱스는 단순 작업 반복·확인·테스트가 끝없이 이어져 매번 확인 버튼을 누르는 게 번거로웠다(= 터미널에서 "허용"을 누르던 피로와 같은 구조). 하지만 반드시 멈춰 확인해야 하는 중대한 단계도 있어, 그걸 자동 통과시키면 개발이 걷잡을 수 없이 커진다. 그래서 질문이 "어떻게 다 자동화하지?"에서 "어디에 게이트를 둘 것인가?"로 바뀌었다. 실측으로 내 전역 `~/.claude/settings.json`은 `{}`(게이트를 의식적으로 설계한 적 없음)인데, 이 저장소엔 `Bash(git check-ignore *)`·`Read(~/.claude/**)`가 allow에 쌓여 "반복은 열고 중대한 건 닫는" 설계의 실물이 보였다. 임정의 '트리거 3종'·아카이브된 IAM 발표안과 안 겹치게 '권한 위임 설계' 축을 골랐다.
