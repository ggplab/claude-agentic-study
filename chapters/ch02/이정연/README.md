## 1. 개발 프로세스로서의 TDD

**개념정리**
TDD를 그동안 논리·경험으로만 체득해 왔는데, 2장은 그것을 하나의 "개발 프로세스"로 개념화한다. 즉 테스트를 먼저 두는 것이 단순 습관이 아니라, 에이전트에게 일을 맡길 때 **검증 가능한 목표를 먼저 고정**해 두는 워크플로의 골격이라는 점이 새로웠다.

**내 시스템 적용예시**
- 아직 이 저장소에 자동 테스트 워크플로는 없음. 다만 이번 작업 흐름 자체가 같은 구조였다 — `_drafts/`에 원본을 먼저 고정(=기대값) → 정리본 생성 → `git check-ignore`로 "초안이 새지 않는다"를 검증하는, 사실상의 "선(先) 기준 설정 후 실행" 패턴.

## 2. settings.json — 무심코 누른 '허용'의 정체

**개념정리**
터미널에서 클로드가 권한을 물을 때 생각 없이 누르던 "yes"는 휘발되는 게 아니라 `.claude/settings.json`(또는 `settings.local.json`)의 `permissions.allow`에 **누적**된다. 즉 설정 파일은 내가 그동안 내려온 권한 결정의 기록이다.

**내 시스템 적용예시 (실측)**
- 전역 `~/.claude/settings.json` = `{}` — 비어 있다. 즉 내 권한은 전역이 아니라 **프로젝트 단위**에 쌓이고 있었다.
- 이 스터디 저장소 `.claude/settings.local.json`에 실제로 두 건이 쌓여 있다:
  ```json
  { "permissions": { "allow": [
      "Bash(git check-ignore *)",
      "Read(//Users/jungyeon/.claude/**)"
  ] } }
  ```
  둘 다 내가 이번 세션 중 "허용"을 누른 결과물이다. 메모에 적은 arxiv 사례(블로그에서 "허용"을 누르자 `WebFetch(domain:arxiv.org)`가 allow에 자동 추가)와 같은 메커니즘을, 이 저장소에서 내 눈으로 다시 확인한 셈.

## 3. cleanupPeriodDays — 클로드는 언제까지 기억하는가

**개념정리**
Resume할 때마다 "얘가 어디까지 기억하지?"가 궁금했는데, 그 보존 범위가 `cleanupPeriodDays` 설정으로 제어된다는 걸 처음 알았다. 세션/대화 기록을 며칠까지 보관하고 그 이후 정리할지를 정하는 값이다.

**내 시스템 적용예시 (실측)**
- 내 `settings.json`에 `cleanupPeriodDays`를 **명시한 적이 없다** → 기본값으로 동작 중.
- 그래도 정리 메커니즘은 실제로 살아 있다: `~/.claude/.last-cleanup` 파일에 마지막 정리 시각 `2026-06-14`이 찍혀 있다. "기억을 무한정 쌓는 게 아니라 주기적으로 비운다"는 걸 파일로 확인.
- 도입한다면: 오래 끄는 프로젝트에서 기록을 더 오래 남기고 싶을 때 전역 `~/.claude/settings.json`에 `"cleanupPeriodDays": <원하는 일수>`를 넣으면 된다.

## 4. 스킬 — 팀의 작업 방식을 코드로 굳히기

**개념정리**
스킬에 "팀 리뷰 방법" 같은 절차를 설정할 수 있다는 점이 인상적이었다. 회사의 리뷰 규칙을 사람마다 다르게 외우는 대신, 호출 가능한 절차로 굳혀 두면 누가 실행하든 같은 흐름이 나온다.

**내 시스템 적용예시 (실측)**
- 추상적인 얘기가 아니라 이미 내 저장소에 구현돼 있었다. `.claude/skills/`에 세 개가 있다: `study-chapter`(지금 이 정리를 만든 스킬), `order-session`(리더가 발표 순서를 조정하는 절차), `chapter-deck`(슬라이드덱 생성).
- 특히 `order-session`이 바로 메모에서 떠올린 "팀 작업 방식의 스킬화"의 실례다 — "일요일 오후에 글을 읽고 중복 점검 후 순서를 제안"이라는 팀 규칙이 스킬 한 개로 박제돼 있다.
- 회사 적용 아이디어: 우리 팀 코드리뷰 체크리스트(네이밍·테스트·로그)를 같은 형식의 스킬로 만들면, 리뷰 호출 한 번으로 동일 기준이 적용된다.

## 5. 확장 도구 5가지 — 무엇을 알고 vs 무엇을 할 수 있는가

> ⚠️ 이 절은 임정 발표 주제("도구 5가지를 트리거 3종으로")와 겹치는 영역이라, 내 발표 핵심으로는 **고르지 않는다**. 개인 이해용으로만 정리.

**개념정리**
- **스킬** = "무엇을 알고 있는가"의 지식 확장.
- **서브에이전트** = "무엇을 할 수 있는가"의 능력 확장. 핵심은 **별도 컨텍스트를 따로 지정**할 수 있다는 점.
- **MCP** = 외부 도구·데이터로 연결하는 통로. (아직 잘 안 써봐서 뒤 예제에서 더 볼 것)
- **훅** = 솔직히 아직 생소. 다음 절에서 별도로.
- **플러그인** = 가장 넓은 컨테이너. 여러 확장 수단을 재사용 가능한 패키지로 묶는 메타 확장이며, 앱처럼 마켓플레이스에 올릴 수 있다.
- 책이 말하는 **가장 중요한 구분은 "실행 주체와 컨텍스트"**다.

**내 시스템 적용예시 (실측)**
- 서브에이전트: `~/.claude/agents`가 비어 있음 → **아직 없음**. 도입한다면, 메모의 블로그 사례대로 "연구 리뷰만 작성하는 서브에이전트"가 1순위(고유 컨텍스트로 논문 리뷰 톤·형식을 고정). `~/.claude/agents/research-review.md`로 만들면 됨.
- 플러그인: 마켓플레이스 `claude-plugins-official` 1개가 등록돼 있고(`~/.claude/plugins/known_marketplaces.json`), 실제 설치된 플러그인은 아직 없음 → "컨테이너는 연결됐지만 안에 담은 건 없는" 상태.

## 6. 훅 — 아직 생소한, 기계적 발화 지점

**개념정리**
훅은 설명을 읽었지만 아직 손에 안 잡힌다. 지금 이해한 선에서는 "특정 이벤트가 발생하면 모델 판단과 무관하게 무조건 실행되는 결정론적 트리거"로 정리해 둔다.

## 7. IAM 권한 관리 — 개발 흐름을 끊지 않는 게이트

**개념정리**
`/permissions`로 권한 설정을 확인했다. 지금 당장 절실하진 않았지만, 권한 관리가 결국 **개발 자동화 흐름에서 "중단"을 최소화하는 게이트 설계**라는 관점이 와닿았다. 어디서 멈춰 물어볼지(allow/deny/ask)를 미리 정해 두는 일이다. 직접 프로젝트를 굴리며 체득할 영역.

**내 시스템 적용예시 (실측)**
- 2절의 `settings.local.json` allow 목록이 바로 내 게이트의 현재 상태다. `Bash(git check-ignore *)`를 allow에 둔 덕에 이번 작업 중 그 명령에서 멈춰 묻지 않았다 — "게이트를 한 번 열어두면 흐름이 안 끊긴다"를 작게 체감.

## 8. 데이터 보존 기간 · 모델

**개념정리**
- 개인 사용자 데이터 **보존 기간이 생각보다 길어서** 놀랐다 (3절의 `cleanupPeriodDays`와 연결되는 지점).
- 모델: 최근 **Fable 5**(메모의 "fables"는 Fable이 맞다 — 정정)가 나와서 만져봤고, `opusplan` 같은 하이브리드 접근(계획 단계와 실행 단계에 서로 다른 모델을 쓰는 방식)이 있다는 게 신기했다.

> ❓ 읽으며 생긴 질문과 내 결론: "opusplan을 왜 갑자기 중단했을까?" → 이건 내가 단정할 수 없는 부분이다. 메모에 링크한 Anthropic 공지(`fable-mythos-access`)와 책 본문을 직접 확인해야 답할 수 있어서, **추정하지 않고 열린 질문으로 남겨 둔다.** (모델 라인업·가용성은 자주 바뀌므로 공식 출처 확인이 정답.)

## ✅ 이번 주 발표용 핵심 개념 1개

**선택: "개발 프로세스 자동화를 위한 '권한 위임 시스템'을 어떻게 설계할 것인가 — 다 자동화하지도, 매번 멈추지도 않는 게이트 디자인"** (위 2·7절을 능동적 설계 질문으로 확장)

**핵심 메시지**
권한 설정의 진짜 쟁점은 "yes를 누를까 말까"가 아니라, **무엇을 자동 통과(allow)시키고 무엇을 반드시 멈춰 확인(ask/deny)하게 둘 것인가를 미리 설계하는 일**이다. 양쪽 극단은 둘 다 실패한다 — 전부 자동화하면 통제를 잃고, 매번 확인하면 흐름이 끊겨 자동화의 의미가 사라진다. 좋은 권한 위임은 이 둘 사이에 게이트를 의도적으로 배치하는 디자인 문제다.

**왜 나에게 중요한가 (로보틱스 현장)**
- 내가 일하는 로보틱스 분야는 **단순 작업 반복 + 확인 + 테스트**가 끝없이 이어진다. 그때마다 확인 버튼을 누르는 게 정말 번거로웠다 — 이건 클로드 터미널에서 매번 "허용"을 누르던 피로와 정확히 같은 구조다.
- 그러나 **반드시 멈춰서 확인하고 넘어가야 하는 중대한 단계**도 분명히 있다. 이걸 자동 통과시키면 개발 과정이 걷잡을 수 없이 복잡해지고 커진다(=안전·되돌릴 수 없는 동작).
- 그래서 질문이 "어떻게 다 자동화하지?"에서 **"어디에 게이트를 둘 것인가?"**로 바뀌었다. allow/ask/deny는 바로 그 게이트 배치를 코드로 적는 언어였다.

**실측 근거 (추정 아님)**
- 내 전역 `~/.claude/settings.json` = `{}` → 게이트를 한 번도 의식적으로 설계한 적이 없었다는 뜻.
- 반면 이 저장소엔 `Bash(git check-ignore *)`·`Read(~/.claude/**)`가 allow에 쌓여 있다 — 반복·안전한 작업은 자동 통과로 굳고, 그 외는 여전히 멈춰 묻는다. "반복은 열고 중대한 건 닫는" 설계의 작은 실물.

**선택 이유**
1. **겹치지 않음**: 임정이 "도구 5가지 → 트리거 3종"을 가져갔고 IAM 발표안은 아카이브했다(`chapters/ch02/임정/`). **권한 위임 설계** 축은 비어 있고, 나는 그걸 "도구 분류"가 아니라 "현장에서의 설계 문제"로 다룬다.
2. **내 분야와 직결**: 로보틱스의 반복 확인 피로 vs 중대 확인 단계라는 실제 고충에서 나온 질문이라, 책 요약이 아니라 내 작업 인사이트가 콘텐츠가 된다.
3. **실측이 단단함**: 빈 전역 설정 vs 프로젝트에 쌓인 allow 2건으로 "게이트 설계의 유무"를 눈으로 보여줄 수 있다.
