# isaac-dev — Isaac Sim 로보틱스 개발용 클로드 플러그인 (사이드 프로젝트 기획서)

> **한 줄 요약**: NVIDIA Isaac Sim 로보틱스 개발에서 반복되는 작업은 자동으로 통과시키고, 되돌릴 수 없는 중대한 단계는 반드시 멈춰 확인하게 만드는 **권한 위임 게이트**를, ch2에서 배운 스킬·서브에이전트·MCP·훅으로 묶은 하나의 클로드 플러그인.
>
> ch2 핵심 개념([README.md](README.md))의 실전 적용 사이드 프로젝트. 이 문서는 기획서이며 구현은 별도.

---

## 1. 배경 & 목표

ch2에서 내가 고른 핵심은 "개발 프로세스 자동화를 위한 권한 위임 시스템(게이트) 설계"였다. 그 추상적 개념을 내 분야인 **로보틱스 시뮬레이터(Isaac Sim)** 개발에 직접 적용해 본다.

- **목표**: Isaac Sim 개발 흐름을 `isaac-dev`라는 클로드 플러그인 하나로 묶어, 반복 작업의 확인 피로를 줄이면서도 중대한 사고(시크릿 커밋, main 강제 푸시 등)는 게이트로 막는다.
- **학습 목표**: 스킬·서브에이전트·MCP·훅 네 가지를 한 프로젝트에서 모두 써 보며 "실행 주체와 컨텍스트"의 차이를 손으로 체득한다. (특히 아직 못 써본 MCP·훅의 첫 실전)
- **대상 범위**: Isaac Sim 중심, IsaacLab-Arena의 Docker·테스트·금지선 패턴을 차용.

분석한 두 공식 가이드:

여기서 보이는 실제 개발 고충:

| # | 고충 | 근거 |
|---|------|------|
| P1 | **모든 명령을 Docker로 감싸야** 한다 (`docker exec "$ARENA_CONTAINER" su ... -c "..."`), 컨테이너명은 동적이라 하드코딩 불가 | Arena |
| P2 | **헤드리스 sim 원격 구동·검증의 반복** (TCP 8226 원격제어, diff-gif 회귀 확인, Tracy 프로파일링) | IsaacSim |
| P3 | **엄격한 금지선을 사람이 매번 지켜야** 한다 (main force-push 금지, AI-attribution 금지, 모델/데이터셋/시크릿 커밋 금지, `docker/`·`workflows`·`submodules` 무단 수정 금지, DCO 서명) | Arena |
| P4 | **정형 절차의 반복** (URDF→USD 변환, SimulationApp inner/outer 테스트 작성, `ArenaEnvBuilder.make_registered()`/`unwrapped` 관례) | 양쪽 |

핵심: P1·P2·P4는 "반복이라 자동화하고 싶은" 영역이고, P3은 "자동화하면 안 되는, 반드시 멈춰야 하는" 영역이다. 이 둘을 가르는 것이 곧 게이트 설계다.

## 3. 솔루션 개요 — "플러그인 = 컨테이너"

ch2에서 배운 대로 **플러그인은 여러 확장 수단을 재사용 가능한 패키지로 묶는 메타 확장(컨테이너)**이다. `isaac-dev` 플러그인 하나에 스킬·서브에이전트·MCP·훅을 담아 마켓플레이스로 배포하면, 팀원 누구나 설치 한 번으로 동일한 Isaac Sim 개발 게이트를 갖게 된다.

```
isaac-dev/  (플러그인 = 컨테이너)
├── skills/        지식 확장      → 정형 절차를 굳힘            (P1, P4)
├── agents/        능력 확장      → 격리 컨텍스트로 무거운 작업  (P2)
├── mcp/           외부 연결      → sim 인스턴스·컨테이너에 연결 (P1, P2)
└── hooks/         결정론적 게이트 → 금지선을 기계적으로 강제     (P3)
```

## 4. 컴포넌트별 설계 (ch2 4도구 매핑)

### 4-1. 스킬 — "무엇을 알고 있는가" (지식 확장)

모델이 설명을 읽고 자율 호출. 정형 절차를 박제한다.

| 스킬 | 역할 | 대응 고충 |
|------|------|-----------|
| `dev-container` | `docker exec "$ARENA_CONTAINER" su $(id -un) -c "cd /workspaces/... && <cmd>"` 래핑 규약 + 동적 컨테이너명 자동 해석 절차 | P1 |
| `urdf-to-usd` | URDF→USD 변환 단계와 검증 포인트 | P4 |
| `arena-env-build` | `ArenaEnvBuilder.make_registered()` 후 `env.unwrapped.cfg/device` 접근 관례 | P4 |
| `sim-test-pattern` | SimulationApp inner/outer 함수 + deferred import 테스트 작성 가이드 | P4 |

각 스킬은 `SKILL.md` 한 개로 "언제 발동 / 무엇을 한다"를 명시. (IsaacSim 레포의 `skills/SKILLS.md` 온디맨드 인덱스 방식 차용)

### 4-2. 서브에이전트 — "무엇을 할 수 있는가" (능력 확장 = 격리 컨텍스트)

별도 컨텍스트를 따로 지정해, 메인 대화를 오염시키지 않고 무거운 작업을 위임.

| 서브에이전트 | 역할 | 대응 고충 |
|--------------|------|-----------|
| `sim-runner` | 헤드리스 Isaac Sim을 TCP 8226으로 구동해 시나리오 실행·캡처 | P2 |
| `validation-triage` | diff-gif를 읽고 회귀 발생 지점·원인 1차 분류 | P2 |
| `profiler` | Tracy 프로파일링 결과를 비교해 프레임 병목 보고 | P2 |

→ 각각 고유 컨텍스트로 분리하는 이유: sim 로그·캡처·프로파일 데이터는 양이 많아 메인 컨텍스트에 두면 금방 넘친다. "능력 확장 = 별도 컨텍스트"라는 ch2 정의의 실증.

### 4-3. MCP — 외부 도구·데이터로의 통로

| MCP 서버 | 노출 도구 | 대응 고충 |
|----------|-----------|-----------|
| `isaac-remote-mcp` | TCP 8226 소켓을 도구로: 씬 로드 / 스텝 실행 / 프레임 캡처 | P2 |
| `arena-container-mcp` | 동적 컨테이너명 해석 후 임의 명령 실행을 안전 도구로 노출 | P1 |

→ 솔직히 나는 아직 MCP를 제대로 써본 적이 없다. 이 프로젝트를 MCP **첫 실전**으로 삼는다. (책 뒤편 MCP 예제를 본 뒤 이 절을 보강)

### 4-4. 훅 — 결정론적 게이트 (이 기획의 핵심)

이벤트에 무조건 발화하는 결정론. **자동화하면 안 되는 중대 단계를 사람 대신 기계가 막는다.** ch2 핵심("반복은 열고 중대한 건 닫는다")이 가장 직접적으로 구현되는 곳.

| 훅 (PreToolUse / pre-commit) | 막는 것 | 근거 |
|------|---------|------|
| `block-force-push-main` | `main`/`release/*` 강제 푸시 차단 | Arena NEVER |
| `block-ai-attribution` | 커밋 메시지의 AI-attribution 라인 차단 (DCO `-s`만 허용) | Arena NEVER |
| `block-secret-commit` | 모델·데이터셋·시크릿 파일 스테이징 차단 | Arena NEVER |
| `warn-protected-paths` | `docker/`·`.github/workflows/`·`.pre-commit-config.yaml`·`submodules/` 수정 시 경고·확인 | Arena NEVER |
| `enforce-container-exec` | Isaac Sim/Arena 명령을 컨테이너 밖에서 직접 실행하려 하면 차단 | Arena P1 |

## 5. ch2 개념 연결

- **실행 주체와 컨텍스트**: 스킬·서브에이전트는 모델이 판단해 호출(자율), 훅은 이벤트에 무조건 발화(결정론), MCP는 그 둘이 닿을 외부 통로. 이 한 프로젝트에 셋이 다 들어간다.
- **권한 위임 게이트**: 반복·안전한 작업(스킬·서브에이전트로 자동 통과 = allow)과 되돌릴 수 없는 중대한 작업(훅으로 멈춤 = deny/ask)을 명시적으로 가른다. 이게 ch2에서 내가 강조하고 싶었던 "다 자동화하지도, 매번 멈추지도 않는" 설계의 실물.

## 6. 설계 원리 — 블래스트 반경(blast-radius) 티어 게이트

이 플러그인의 컴포넌트들은 "스킬/서브에이전트/MCP/훅"이라는 **도구 종류**로 나누기보다, **되돌릴 수 없는 정도 = 블래스트 반경**으로 묶는 게 내 ch2 핵심("중대한 건 멈추고 반복은 연다")에 더 맞다. 로보틱스에서 '중대함'의 진짜 척도는 한 동작이 사고 시 미치는 범위이기 때문이다.

| 티어 | 작업 예 | 블래스트 반경 | 게이트 정책 | 해당 컴포넌트 |
|:---:|------|------|------|------|
| **T0** | sim 내부 편집(씬 트윅, 로그 조회) | 0 (sim 안에서 끝남) | 자동 통과 (allow) | 스킬 전반 |
| **T1** | 에셋·물리 상수 변경(USD, 마찰, 게인) | 다운스트림 전체 | 감사·diff 통과 시 통과 (ask) | `urdf2usd`(B), `sim-physics-tuner`(H) |
| **T2** | 학습/데이터셋 산출물 커밋 | 팀 공유물 오염 | 회귀 검증 게이트 (ask/deny) | `sim-regression-guard`(F), 물리 회귀 CI(J) |
| **T3** | **실제 로봇 배포** | 물리적 안전·장비 | **하드 스톱 — 사람 승인 필수** (deny) | `sim2real-deploy-gate`(G) |

→ 이렇게 보면 흩어진 후보들이 **하나의 척도 위에 놓인 게이트 단계**가 된다. 같은 훅이라도 T1은 "통과시키되 기록", T3은 "무조건 멈춤"으로 정책이 달라진다. "왜 이 구성인가"에 답을 주는 조직 원리.

## 7. MVP 단계 로드맵

1. **1단계 (효용 최대)**: `dev-container` 스킬 + 금지선 훅 4종. 컨테이너 래핑 자동화 + 사고 방지 게이트만으로도 일상 흐름이 크게 편해진다.
2. **2단계**: `sim-runner` 서브에이전트 + `isaac-remote-mcp`. 헤드리스 sim 구동·캡처를 위임.
3. **3단계**: `validation-triage`·`profiler` + diff-gif/Tracy 자동화. 회귀·성능 검증까지.

## 8. 성공 기준 / 검증

- **편의**: 일상 작업에서 사람이 누르는 "확인" 횟수가 줄어든다 (반복 게이트가 열림).
- **안전**: 시크릿/모델 커밋, main 강제 푸시 같은 중대 사고 0건 (중대 게이트가 닫힘).
- 이 두 지표를 **동시에** 만족하면 게이트 설계가 성공한 것 — 한쪽만 달성하면 과자동화 또는 과확인.

| J | **physics-regression-ci** | 파라미터 변경 시 물리 프로브 자동 실행 + last-known-good diff | 서브에이전트·훅 | ★★★ | 중상 |
| K | **failure-replay** | sim 실패 시 최소 재현 + 근본원인 후보 리포트 | 서브에이전트·MCP | ★★ | 중상 |

> 메모: 후보는 **블래스트 반경 티어(6절)** 위에서 읽는다 — B·H·I는 T1, F·J는 T2, G는 T3, K는 진단 보조.
> - **솔로 MVP 1순위**: **B(urdf2usd)** — 범위가 좁고 결과가 눈에 보임. 또는 **J(physics-regression-ci)** — B·F·H를 CI 한 줄로 묶어 일관적이고 프로브가 스크립트라 혼자 완결 가능.
> - **야심 버전**: **I(real2sim-calibrator)** — H의 약점(실물 없이 튜닝하면 "안정적이지만 비현실적")을 정면 보완. 실물 로그 접근이 전제.
> - **파이프라인**: B(유효한 USD) → H/I(물리적으로 올바르게) → J(회귀 방지) → G(배포 게이트). K는 전 단계의 실패를 진단.
> - C·D·E는 범위가 크고 테마 적합도가 낮아 후순위로 명시.

## 11. 우선 MVP 후보 상세 — `urdf2usd` (변환기가 아니라 가드레일)

> **핵심 전제**: Isaac Sim에는 이미 URDF→USD 임포터(`isaacsim.asset.importer.urdf`, 구 `omni.importer.urdf`)와 Isaac Lab의 `urdf_converter`가 **내장**돼 있다. 변환 자체는 끝난 기능이므로 재구현하지 않는다. **만들 가치는 임포터가 "성공"이라고 말하지만 결과물이 조용히 틀린 경우를 잡아내는 데 있다.**

**한 줄**: 내장 임포터를 호출하되, 그 앞뒤에 게이트를 붙여 **조용한 변환 실패**를 린트·감사·차단한다. URDF 던지면 → 사전 린트 → 임포터 호출 → 결과 USD 감사 + 이전 버전 diff → 통과해야 커밋.

**왜 변환기가 아니라 가드레일인가 (조사 근거)**
임포터는 변환에 "성공"해도 다음을 조용히 틀린다 — 사람이 sim을 돌려본 뒤에야 발견하는 게 반복 루프 최대의 낭비다.

| 조용한 실패 | 증상 | 근거 |
|------------|------|------|
| Mimic 조인트 | finite limit 없으면 실패, 되더라도 스프링처럼 진동 | [Issue #305](https://github.com/isaac-sim/IsaacSim/issues/305), [포럼](https://forums.developer.nvidia.com/t/mimic-joints-are-lagging-behind-parent-joints/330070) |
| 머티리얼 유실(회귀) | 5.1 ROS2 임포트에서 Looks가 텅 빔(5.0은 정상) | [Issue #300](https://github.com/isaac-sim/IsaacSim/issues/300) |
| fixed joint 병합 변경 | mass/inertia 있는 링크는 `merge-joint=True`여도 병합 안 됨 → 바디 수 변동 | [Isaac Lab 마이그레이션](https://isaac-sim.github.io/IsaacLab/main/source/refs/migration.html) |
| stale USD | 메시만 바꾸면 USD 재생성 안 됨(lazy) → 옛 USD 사용 | [Issue #885](https://github.com/isaac-sim/IsaacLab/issues/885) |
| Convex Decomposition | 플래그 토글해도 출력 동일, meshSimplification 콜라이더 사용 불가 | [포럼](https://forums.developer.nvidia.com/t/importing-pre-computed-convex-decomp-collision-meshes-from-urdf/202562) |
| 경로 네이밍 | 하이픈 자동 치환 → 다운스트림 참조 깨짐 | 포럼 |
| planar 조인트 / 센서 | planar 미지원, 센서는 camera·lidar만 파싱 | [공식 문서](https://docs.isaacsim.omniverse.nvidia.com/6.0.0/importer_exporter/import_urdf.html) |

**세 가지 능력**
1. **Pre-flight 린트(변환 전)** — URDF를 미리 스캔해 임포터가 망칠 것을 경고·자동수정 제안: planar 조인트, limit 없는 mimic, 하이픈 이름, 관성·질량 0, 풀리지 않는 `package://`.
2. **Post-import 감사 + diff(변환 후)** — 결과 USD 점검(Looks 비었나, fixed-joint 병합으로 바디 수 변했나, 콜라이더가 진짜 convex decomp인가) + **이전 USD와 비교**해 머티리얼 유실 같은 회귀 검출.
3. **반복 루프 도우미** — 메시 해시를 추적해 stale USD 감지·강제 재변환.

**컴포넌트 매핑 (ch2 4도구)**
| 도구 | 이름 | 역할 |
|------|------|------|
| 스킬 | `urdf-preflight` | 변환 전 린트 체크리스트 + 임포터 호출 규약 |
| 서브에이전트 | `usd-auditor` | 변환 후 USD 감사 + 이전 버전 diff 리포트(로그 대량 → 격리 컨텍스트) |
| MCP | `isaac-remote-mcp` | 라이브 로드해 시각 검증·스크린샷 |
| 훅 | `block-broken-usd-commit` | 감사 미통과·회귀 USD의 커밋 차단(중대 게이트) |

**왜 이걸 1순위 MVP로**
- 입력·출력이 명확하고(URDF in → 린트/감사 리포트 + 검증된 USD out) 외부 sim 의존이 적어 **혼자 완결 가능**.
- "성공이라 떴는데 틀린" 문제를 잡으므로 가치가 즉시 체감되고 데모가 명확하다.

**성공 기준**
- 변환 후 sim을 돌려보고서야 버그를 찾는 횟수 감소(반복 게이트가 열림).
- 머티리얼 유실·stale USD 같은 회귀가 main에 들어가는 사고 0건(중대 게이트가 닫힘).
- → 본문 8절의 "편의 + 안전 동시 충족" 기준을 urdf2usd 범위에서 검증.

## 12. 확장 아이디어 — `sim-physics-tuner` (제어기 선택 + 물리 상수 튜닝)

> 유효한 USD가 나와도(11절) **물리적으로 올바르게 움직이는 건 별개**다. 어떤 제어기를 넣을지, 마찰·충돌·강성 같은 상수를 얼마로 둘지는 정답이 없고 **돌려보며 맞추는 경험적 작업**이라 가장 번거롭다. `urdf2usd`(B) → `sim-physics-tuner`(H)가 자연스러운 파이프라인.

**왜 어려운가 (문제의 본질)**
- **제어기 선택**: 위치/속도/토크 드라이브, PD 게인(stiffness·damping), 임피던스/OSC 등 — 로봇 종류(암·쿼드러페드·모바일)와 태스크에 따라 적합한 게 다르다.
- **물리 상수**: 정·동마찰, 반발계수(restitution), 접촉 강성·감쇠, joint armature, 최대 속도, 솔버 반복수 — 값 하나가 안정성·sim2real 갭을 좌우한다.
- 정답이 없어 **수동 시행착오 루프**(값 바꾸고 → 돌리고 → 출렁임·관통·불안정 관찰 → 다시)가 길다.

**핵심 아이디어 — 닫힌 루프 튜닝 보조**
사람이 값을 찍어 넣는 대신, 에이전트가 sim 실험을 돌려 **측정 → 제안 → 재실험**을 반복한다.
1. **제어기 추천(스킬)**: 로봇 형태·관절 수·태스크(매니퓰레이션/보행/주행)를 입력하면 시작 제어기 + 초기 게인 프리셋 제시.
2. **상수 스윕(서브에이전트 + MCP)**: `physics-sweeper`가 마찰·강성·감쇠 등을 격자/베이지안으로 스윕하며 헤드리스 sim을 다회 실행, **추종 오차·관통·발진**을 측정해 리포트(대량 trial → 격리 컨텍스트).
3. **안정성 게이트(훅)**: 발진·관통·NaN을 내는 파라미터 셋은 **채택 차단**. "불안정한 게인을 실수로 커밋/배포"하는 중대 실패를 막는다 → ch2 테마와 직결.

**컴포넌트 매핑 (ch2 4도구)**
| 도구 | 이름 | 역할 |
|------|------|------|
| 스킬 | `controller-advisor` | 로봇·태스크 → 제어기·초기 게인 프리셋 추천 규약 |
| 서브에이전트 | `physics-sweeper` | 마찰·강성·감쇠 등 상수 스윕 + 안정성/추종오차 측정 리포트 |
| MCP | `isaac-remote-mcp` | 각 파라미터 셋을 라이브 sim에 적용·실행·계측 |
| 훅 | `block-unstable-params` | 발진·관통·NaN 유발 파라미터 채택/커밋 차단 |

**측정 지표(예시)**
- 추종 오차(목표 vs 실제 관절 궤적), 정착 시간·오버슈트
- 접촉 관통 깊이, 비물리적 에너지 증가(발진), NaN/폭발 여부
- (가능 시) 실물 데이터와의 sim2real 갭

**성공 기준**
- 수동 튜닝 반복 횟수·시간 감소(반복은 자동화로 열림).
- 불안정 파라미터가 채택·배포되는 사고 0건(중대한 건 멈춤).
- → urdf2usd와 동일하게 8절의 "편의 + 안전 동시 충족"으로 평가.

## 13. 확장 아이디어 — `real2sim-calibrator` (reality gap 줄이기 · H의 야심 버전)

> H(`sim-physics-tuner`)의 약점: 실물 데이터 없이 튜닝하면 "안정적이지만 현실과 다른" sim이 나온다. 로보틱스의 진짜 고통은 **reality gap**이므로, 튜닝의 기준을 sim 내부 안정성이 아니라 **실물과의 일치**로 바꾼다. 티어 T1.

**핵심 아이디어 — 실물 로그를 정답으로 둔 시스템 식별**
실제 로봇에서 같은 명령을 준 로그(rosbag 등)와 sim 로그를 정렬하고, 둘의 차이를 최소화하는 물리 상수(마찰·강성·감쇠·armature 등)를 역으로 추정한다. 즉 "값을 찍어보는" 게 아니라 "데이터가 값을 정한다".

**컴포넌트 매핑 (ch2 4도구)**
| 도구 | 이름 | 역할 |
|------|------|------|
| 스킬 | `log-align` | 실물/ sim 로그 시계열 정렬·동기화 규약 |
| 서브에이전트 | `sysid-optimizer` | gap을 목적함수로 물리 상수 최적화(다회 sim 실행 → 격리 컨텍스트) |
| MCP | `isaac-remote-mcp` | 후보 파라미터를 sim에 적용·재생·계측 |
| 훅 | `block-overfit-params` | 특정 로그에만 맞춘 과적합·물리적으로 비현실적인 값 채택 차단 |

**성공 기준**
- 동일 명령에 대한 실물 vs sim 궤적 오차(gap) 감소가 정량적으로 보일 것.
- 과적합·비현실적 파라미터가 채택되는 사고 0건.
- **전제/리스크**: 실물 로그 접근과 신뢰할 캘리브레이션 데이터가 필요(난이도 상).

## 14. 확장 아이디어 — `physics-regression-ci` (가장 solo-MVP다운 통합형)

> B·F·H를 **CI 한 줄**로 묶는 형태. 에셋·물리 파라미터가 바뀔 때마다 정해진 물리 프로브를 자동 실행해 **last-known-good 시그니처와 diff**한다. "누가 마찰을 바꿨더니 파지가 깨짐"을 사람이 아니라 CI가 잡는다. 티어 T2.

**물리 프로브(예시)**
- 드롭 테스트: 정지 자세에서 낙하·안착 — 관통/튕김 없는가
- 관절 스윕: 각 관절을 가동범위 끝까지 — 한계·토크 이상 없는가
- 접촉 테스트: 기준 물체 파지/접촉 — 마찰로 미끄러짐·관통 없는가
- 각 프로브 결과를 수치 시그니처(안착 높이, 접촉력, 미끄럼량 등)로 저장

**컴포넌트 매핑 (ch2 4도구)**
| 도구 | 이름 | 역할 |
|------|------|------|
| 스킬 | `probe-suite` | 표준 물리 프로브 정의·실행 규약 |
| 서브에이전트 | `regression-judge` | 새 시그니처 vs last-known-good diff·원인 1차 분류 |
| 훅 | `block-physics-regression` | 시그니처 회귀(허용 오차 초과) 시 머지/커밋 차단 |

**왜 매력적인가**
- 프로브가 전부 스크립트라 **혼자 완결 가능**하고, 한 번 만들면 B·H·I 산출물 모두에 재사용된다.
- "성공이라 떴는데 물리가 조용히 바뀐" 회귀를 정면으로 막아 테마 적합도 높음.

## 15. 확장 아이디어 — `failure-replay` (실패 리플레이 & 근본원인)

> 로보틱스 시간의 대부분은 "왜 실패했지"를 추적하는 데 쓰인다. sim 에피소드 실패(낙상·파지 실패·발산) 시 상태를 캡처해 **최소 재현 + 근본원인 후보**를 돌려준다. 게이트라기보다 진단 보조(전 단계의 실패를 받아 처리).

**핵심 아이디어**
- 실패 직전 N스텝의 상태(관절·접촉·명령)를 스냅샷으로 캡처
- 결정론적 시드로 **최소 재현 스크립트** 생성
- 원인 후보 랭킹(예: 접촉 솔버 발산 vs 게인 과대 vs 충돌 메시 오류) + 근거 로그 인용

**컴포넌트 매핑 (ch2 4도구)**
| 도구 | 이름 | 역할 |
|------|------|------|
| 스킬 | `repro-capture` | 실패 시점 스냅샷·최소 재현 생성 규약 |
| 서브에이전트 | `root-cause` | 로그·상태를 읽어 원인 후보 랭킹 리포트(로그 대량 → 격리 컨텍스트) |
| MCP | `isaac-remote-mcp` | 재현 스크립트를 sim에 재생해 가설 검증 |

**성공 기준**
- 실패당 평균 디버깅 시간 단축, "재현 안 됨"으로 흘려보내는 실패 비율 감소.
- 다른 후보(특히 J·I)와 연동: 회귀·gap이 감지되면 `failure-replay`가 원인 추적을 이어받음.
