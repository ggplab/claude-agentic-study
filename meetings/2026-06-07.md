## 핵심 논의

### 1. 운영 방식 확정
- 어려운 기술책은 **오너십**(내가 이 단원 리드)과 **너무 깊게 안 가기**가 핵심. 1시간 스터디에 4~5시간 투입하는 방식은 지양 (다들 본업 있음).
- 총 10장 / 참여자 5명 → **사다리타기로 1인당 2장씩 배정**. 마지막 장은 게임 만들기.
- 문종운 제안 채택: 각 장마다 **자기에게 필요한 걸 하나씩 만들어 보고 시행착오를 공유**.
- 주차별 **리드**의 역할: 각자 써온 주제를 책의 절 순서에 맞춰 **발표 순서 교통정리** (겹치는 주제 협의, 깊이 다르면 간단한 것 먼저). 결석 시 리드 인수인계.
- 시간대: **일요일 밤 10시** 합의.

### 2. 결과물 정의 + 아카이빙
- **주간 결과물 = 챕터 글 작성 + LinkedIn 포스팅까지** (임정은 스레드 주간 챌린지 병행). 전원 포스팅 동의.
- GitHub repo는 **아카이빙용**, 챕터별로 각자 시행착오·생각을 기록.
- PR 받지 않고 **메인 브랜치에 직접 머지**하기로 (전원 git 사용 가능 확인).

### 3. 1장 공유: 각자 인상 깊었던 개념
- **문종운**: 클로드 코드의 혁신(능동적 탐색·읽기·편집). 에이전트/서브에이전트/스킬/커맨드를 어떻게 엮는가가 곧 **하네스**. SDD(스펙 주도)·TDD가 잘 맞았다는 설명. **체크포인트·rewind** 기능을 새로 알게 됨(세션 저장기간 조절, 이전 대화/코드 복구). 책엔 컨텍스트 20만으로 나오나 현재는 100만 토큰.
- **임정**: 서드파티 안 쓰는 "순정파". 코딩 외 계약서·문서·발표자료까지 처리. **훅(hook)** 애용 → 세션 종료 훅으로 대화·토큰 사용기록을 캘린더에 저장. 훅은 결정론적이라 **재현성** 확보 수단(에이전트 시대 중요).
- **문종운(추가)**: 하네스 변천사. oh-my-claude-code / MoAI-ADK를 써봤지만 통제를 벗어나 자체 하네스로 회귀. 클로드맵(메모리)+옵시디언 온톨로지로 관리. 회사에선 PR 머지 시 위키 수정·코드리뷰 자동 훅 운영.
- **남서아**: **헤드리스 모드(`claude -p`)** 를 처음 알게 됨. resume/continue로 세션 선택 복원이 유용.
- **최지환**: 비개발자. lovable·booster 등과 연동해 개인 프로젝트에 사용. TDD 등 방법론을 더 공부할 필요를 느낌. 다양한 직군의 활용법을 배우고 싶음.
- **이정연**: 앞 의견에 대체로 공감. 잘 안 쓰던 명령어를 알게 됨. 1장은 인트로로 잘 배움.

### 4. 곁가지 토론
- **rewind vs git worktree** 차이: rewind는 클로드의 read/edit/write 도구 작업만 되돌리며 bash·외부 API 호출은 복구 불가. worktree는 세션별 브랜치 분리 개념.
- **`claude -p` 유료화 우려**: 6/15경 구독 외 별도 과금 적용 예정이라 자유로운 사용이 어려워질 수 있음. 우회 방법 고민 중.
